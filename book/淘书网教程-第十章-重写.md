# 第10章：商品列表与分页

## 🤔 为什么需要分页？

> **小王的疑惑**：把所有书都显示出来不就行了吗？一页展示不完再翻页不就行了？

### 从一个性能灾难说起

**场景：网站有10万本书，一次性加载**

```
没有分页的后果：
┌─────────────────────────────────────┐
│ 前端：一次性加载10万本书            │
│ → 数据库查询：SELECT * FROM book   │
│ → 网络传输：10MB JSON数据          │
│ → 浏览器渲染：10万个DOM元素       │
│ → 内存占用：500MB+                 │
└─────────────────────────────────────┘

用户体验：
→ 页面加载时间：30秒+
→ 滚动卡顿
→ 内存溢出崩溃
→ 流量消耗巨大
```

**更糟糕的场景**：

```
恶意行为：
→ 爬虫一次性请求所有数据
→ 服务器负载激增
→ 数据库连接耗尽
→ 正常用户无法访问

拒绝服务攻击：
→ 攻击者发送大量请求
→ 每个请求都查询10万条数据
→ 数据库CPU 100%
→ 服务器宕机
```

---

### 🎯 分页的价值

```
分页后：
┌─────────────────────────────────────┐
│ 第1页（20本书）                      │
│ → 查询：LIMIT 0, 20                  │
│ → 传输：20KB                            │
│ → 渲染：20个DOM元素                    │
│ → 加载时间：0.5秒                       │
└─────────────────────────────────────┘

用户体验：
→ 加载快
→ 流畅浏览
→ 按需加载（翻页时加载更多）
→ 服务器负载均衡
```

### 10.1 分页基础原理

#### 🤔 分页是怎么实现的？

**Python/Django的实现**：

```python
# Django: Paginator
from django.core.paginator import Paginator

def book_list(request):
    books = Book.objects.all()  # 10万本书
    paginator = Paginator(books, 20)  # 每页20本

    page = request.GET.get('page', 1)
    books_page = paginator.page(page)

    return Response({
        'count': paginator.count,
        'page': page,
        'page_size': 20,
        'total_pages': paginator.num_pages,
        'results': BookSerializer(books_page, many=True).data
    })

# SQL: LIMIT 20 OFFSET 0 (第1页)
# SQL: LIMIT 20 OFFSET 20 (第2页)
```

**Java/Spring Boot的实现**：

```java
@GetMapping("/books")
public Result<Page<BookResponse>> getBooks(
        @RequestParam(defaultValue = "1") Integer page,
        @RequestParam(defaultValue = "10") Integer size) {

    // 1. 创建分页请求
    Pageable pageable = PageRequest.of(page - 1, size,
        Sort.by(Sort.Direction.DESC, "createdAt"));

    // 2. 查询数据
    Page<Book> books = bookRepository.findAll(pageable);

    // 3. 转换为DTO
    Page<BookResponse> responses = books.map(BookResponse::fromEntity);

    return Result.success(responses);
}

// SQL: LIMIT 10 OFFSET 0 (第1页)
// SQL: LIMIT 10 OFFSET 10 (第2页)
```

**关键差异**：

| 特性 | Django | Spring Boot |
|------|--------|-------------|
| **页码** | 从1开始 | 从0开始（需要-1） |
| **分页对象** | `Page` | `Pageable` |
| **计数** | `paginator.count` | `page.getTotalElements()` |
| **总页数** | `paginator.num_pages` | `page.getTotalPages()` |

### 10.2 为什么不能用List< Book>？

**问题：用List会怎样？**

```java
// ❌ 错误做法
@GetMapping("/books")
public Result<List<BookResponse>> getAllBooks() {
    List<Book> books = bookRepository.findAll();  # 查询10万本

    // 问题1：内存占用
    // → 10万个对象在内存中
    // → 每个对象约1KB
    // → 总共100MB内存

    // 问题2：网络传输
    // → 100MB JSON数据
    // → 传输时间30秒+

    // 问题3：前端渲染
    // → 渲染10万个DOM
    // → 浏览器卡死

    return Result.success(
        books.stream()
            .map(BookResponse::fromEntity)
            .collect(Collectors.toList())
    );
}
```

**对比Page的优势**：

```
用Page：
→ 只查询当前页的数据（20条）
→ 内存占用：2MB
→ 网络传输：2KB
→ 前端渲染：快

用List：
→ 查询全部数据（10万条）
→ 内存占用：100MB
→ 网络传输：100MB
→ 前端渲染：慢
```

### 10.3 分页参数设计

#### 🤔 为什么页码从0开始？

**Python/Django**：

```python
# Django: page从1开始
page = request.GET.get('page', 1)

# 查询第1页：
# LIMIT 20 OFFSET 0

# 查询第2页：
# LIMIT 20 OFFSET 20
```

**Java/Spring Boot**：

```java
// Java: page从0开始
@RequestParam(defaultValue = "1") Integer page

// 内部处理：page - 1
Pageable pageable = PageRequest.of(page - 1, size);

// 查询第1页：page=1 → page-1=0 → OFFSET 0
// 查询第2页：page=2 → page-1=1 → OFFSET 1
```

**为什么Java要-1？**

```
Spring Data JPA的设计：
→ page从0开始（更符合程序员习惯）
→ page=0表示第1页
→ page=1表示第2页

但用户习惯从1开始：
→ 所以前端传1，后端减1

这样设计的好处：
→ 程序员用0索引（数组、List都是0开始）
→ 用户用1序（人类习惯）
→ 各自方便
```

### 10.4 分页信息设计

#### 🎯 前端需要哪些分页信息？

```
完整分页信息应该包含：
┌──────────────────────────────────┐
│ 第1页 / 共5页                       │
│ 每页20条，共100条记录              │
│ « 1 2 3 4 5 ... 9 10 »        │
└──────────────────────────────────┘

信息分解：
→ 当前页：current_page
→ 每页大小：page_size
→ 总记录数：total
→ 总页数：total_pages
→ 上一页：previous
→ 下一页：next
```

**Spring Boot的Page对象**：

```java
@Data
public class PageResponse<T> {
    private List<T> content;          // 当前页数据
    private int currentPage;          // 当前页码
    private int pageSize;              // 每页大小
    private long totalElements;         // 总记录数
    private int totalPages;            // 总页数
    private boolean first;               // 是否第一页
    private boolean last;                // 是否最后一页
    private boolean empty;               // 是否为空
}
```

### 10.5 排序功能

#### 🤔 为什么需要多种排序方式？

**场景：用户不同的浏览习惯**

```
习惯1：按最新
→ 刚上架的书优先
→ 新书推荐

习惯2：按价格
→ 最便宜的书优先
→ 优惠专区

习惯3：按销量
→ 最受欢迎的书
→ 热销榜单

习惯4：按好评
→ 评分高的书
→ 精品推荐
```

**实现**：

```java
@GetMapping("/books")
public Result<Page<BookResponse>> getBooks(
        @RequestParam(required = false) String sort,
        @RequestParam(defaultValue = "1") Integer page,
        @RequestParam(defaultValue = "10") Integer size) {

    // 创建排序
    Sort sort = Sort.by(Sort.Direction.DESC, "createdAt");  // 默认最新

    // 根据参数调整排序
    if ("price_asc".equals(sort)) {
        sort = Sort.by(Sort.Direction.ASC, "price");
    } else if ("price_desc".equals(sort)) {
        sort = Sort.by(Sort.Direction.DESC, "price");
    } else if ("sales".equals(sort)) {
        sort = Sort.by(Sort.Direction.DESC, "soldQuantity");
    }

    // 创建分页请求
    Pageable pageable = PageRequest.of(page - 1, size, sort);

    // 查询
    Page<Book> books = bookRepository.findAll(pageable);

    return Result.success(books.map(BookResponse::fromEntity));
}
```

### 10.6 搜索功能

#### 🤔 为什么搜索不能用LIKE %keyword%？

**场景：全模糊搜索的性能问题**

```sql
-- 全模糊搜索
SELECT * FROM book
WHERE title LIKE '%Java%'
   OR author LIKE '%Java%'
   OR description LIKE '%Java%'

问题：
→ 即使有索引也用不上
→ %%导致索引失效
→ 全表扫描
→ 10万条数据，查询需要3秒
```

**为什么%%导致索引失效？**

```
B树索引：
┌─────────────────────┐
│     Java          │
└─────────────────────┘
     ↓
┌─────────────────────┐
│     Programming │
└─────────────────────┘
     ↓
...

当使用 '%Java%' 时：
→ 必须从第一个字符开始匹配
→ 无法跳过不匹配的前缀
→ 必须扫描整棵索引树
→ 索引失效
```

---

#### ✅ 搜索优化方案

**方案1：全文索引**

```sql
-- 创建全文索引
ALTER TABLE book ADD FULLTEXT INDEX idx_fulltext (title, author, description);

-- 使用全文索引
SELECT * FROM book
WHERE MATCH(title, author, description) AGAINST('Java' IN NATURAL LANGUAGE MODE);
```

**方案2：Elasticsearch**

```
架构：
┌─────────────┐
│  应用服务器 │
└─────────────┘
      ↓
┌─────────────┐
│  ElasticSearch│  ← 专门的搜索引擎
└─────────────┘
      ↓
┌─────────────┐
│   MySQL     │
└─────────────┘

流程：
1. 书籍数据 → 同步到Elasticsearch
2. 用户搜索 → 查询Elasticsearch
3. 返回书籍ID
4. 从MySQL获取完整信息
```

**方案3：左右模糊（简化版）**

```java
// Repository
@Query("SELECT b FROM Book b WHERE " +
           "b.title LIKE CONCAT('%', :keyword, '%') OR " +
           "b.author LIKE CONCAT('%', :keyword, '%')")
Page<Book> searchBooks(@Param("keyword") String keyword, Pageable pageable);

// MySQL会优化这种情况
// 使用索引跳跃扫描
```

### 10.7 筛选功能

#### 🤔 为什么需要高级筛选？

**场景：用户想找"清华大学出版社2023年出版的计算机类书籍"**

```
基本搜索：
→ 关键词："清华大学" OR "计算机"
→ 结果：1000本书

高级筛选：
→ 出版社：清华大学
→ 出版年份：2023年
→ 分类：计算机
→ 价格：0-50元
→ 成色：全新或九成新
→ 结果：10本书

效率提升：
→ 准少浏览量
→ 快速找到目标
→ 提升转化率
```

**筛选接口**：

```java
@GetMapping("/books/search")
public Result<Page<BookResponse>> searchBooks(
        @RequestParam(required = false) String keyword,
        @RequestParam(required = false) Long categoryId,
        @RequestParam(required = false) String publisher,
        @RequestParam(required = false) Integer year,
        @RequestParam(required = false) BigDecimal minPrice,
        @RequestParam(required = false) BigDecimal maxPrice,
        @RequestParam(required = false) Integer condition,
        @RequestParam(defaultValue = "1") Integer page,
        @RequestParam(defaultValue = "10") Integer size) {

    // 使用Specification动态构建查询条件
    Specification<Book> spec = (root, query, cb) -> {
        List<Predicate> predicates = new ArrayList<>();

        // 关键词搜索
        if (keyword != null && !keyword.isEmpty()) {
            String likeKeyword = "%" + keyword + "%";
            predicates.add(cb.or(
                cb.like(root.get("title"), likeKeyword),
                cb.like(root.get("author"), likeKeyword),
                cb.like(root.get("description"), likeKeyword)
            ));
        }

        // 分类筛选
        if (categoryId != null) {
            predicates.add(cb.equal(root.get("categoryId"), categoryId));
        }

        // 出版社筛选
        if (publisher != null && !publisher.isEmpty()) {
            predicates.add(cb.like(root.get("publisher"), "%" + publisher + "%"));
        }

        // 价格范围
        if (minPrice != null) {
            predicates.add(cb.greaterThanOrEqualTo(root.get("price"), minPrice));
        }
        if (maxPrice != null) {
            predicates.add(cb.lessThanOrEqualTo(root.get("price"), maxPrice));
        }

        // 成色筛选
        if (condition != null) {
            predicates.add(cb.equal(root.get("condition"), condition));
        }

        // 只查上架的
        predicates.add(cb.equal(root.get("status"), 1));

        return cb.and(predicates.toArray(new Predicate[0]));
    };

    // 执行查询
    Specification<Book> spec = Specification.where(spec);
    Pageable pageable = PageRequest.of(page - 1, size,
        Sort.by(Sort.Direction.DESC, "createdAt"));

    Page<Book> books = bookRepository.findAll(spec, pageable);

    return Result.success(books.map(BookResponse::fromEntity));
}
```

### 10.8 缓存策略

#### 🤔 为什么需要缓存热门书籍列表？

**场景：热门列表被频繁访问**

```
每秒10个用户访问热门书籍列表
→ 每次都查询数据库
→ 数据库CPU: 80%
→ 响应时间: 500ms

加上缓存：
→ 热门列表存Redis
→ 缓存命中: 90%
→ 数据库CPU: 20%
→ 响应时间: 50ms
```

**Redis缓存实现**：

```java
@Service
@RequiredArgsConstructor
public class BookService {

    @Cacheable(value = "hotBooks", key = "'hot:' + #page", unless = "#result.isEmpty()")
    public Page<Book> getHotBooks(Integer page, Integer size) {
        Pageable pageable = PageRequest.of(page - 1, size,
            Sort.by(Sort.Direction.DESC, "soldQuantity"));

        return bookRepository.findHotBooks(pageable);
    }

    @CacheEvict(value = "hotBooks", allEntries = true)
    public void clearHotBooksCache() {
        // 当有新书上架或下架时，清空缓存
    }
}

// 缓存配置
@Configuration
@EnableCaching
public class CacheConfig {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {

        // 热门书籍缓存1小时
        RedisCacheConfiguration hotBooksCache = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofHours(1))
            .disableCachingNull();

        return RedisCacheManager.builder(factory)
            .cacheDefaults(hotBooksCache)
            .build();
    }
}
```

### 10.9 无限滚动 vs 页码跳转

#### 🤔 两种翻页方式对比

**方式1：页码跳转**

```
显示：
« 1 2 3 ... 9 10 »

优点：
→ 可以快速跳到任意页
→ 适合总页数少的场景

缺点：
→ 页码很多时显示不下
→ 移动端体验差
```

**方式2：无限滚动**

```
显示：
↓ 继续向下滚动
↓
↓
↓
自动加载更多

优点：
→ 用户体验流畅
→ 适合移动端
→ 符合现代App习惯

缺点：
→ 无法跳转到特定页
→ 返回查看不方便
```

**推荐方案**：

```
PC端：页码跳转 + 无限滚动
→ 首页显示1-10页码
→ 滚动到底部自动加载下一页

移动端：无限滚动
→ 滚动到底部自动加载
→ 顶部有"回到顶部"按钮
```

### 10.10 本章小结

#### 🎯 分页核心要点

| 概念 | 说明 | 最佳实践 |
|------|------|----------|
| **分页原因** | 性能优化 | 避免一次加载过多数据 |
| **Page vs List** | Page按需查询 | List加载全部数据 |
| **页码从0开始** | Java程序员习惯 | 前端从1开始，后端-1 |
| **排序** | 满足多种浏览习惯 | 提供多种排序选项 |
| **搜索** | 全模糊搜索性能差 | 用Elasticsearch或全文索引 |
| **缓存** | 减少数据库负载 | 热门数据存Redis |

#### 💡 性能优化技巧

```
1. 限制每页大小：10-50条
2. 使用索引覆盖常见查询
3. 热门数据用Redis缓存
4. 搜索用专业搜索引擎
5. 避免SELECT *，只查询需要的字段
```
