# 第十九章：消息队列

> **小王的疑问**：用户下单后需要通知库存服务、物流服务、积分服务...如果这些服务挂了，订单还能创建成功吗？如何解耦这些服务？

---

## 🤔 为什么需要消息队列？

### 真实场景：服务耦合的灾难

```
某电商平台没有消息队列：
┌─────────────────────────────────────────────────────────────┐
│  用户下单                                                   │
│      ↓                                                      │
│  订单服务调用：                                             │
│  ├─ 库存服务（扣减库存）                                    │
│  ├─ 物流服务（创建运单）                                    │
│  ├─ 积分服务（增加积分）                                    │
│  ├─ 通知服务（发送短信）                                    │
│  └─ 数据服务（记录日志）                                    │
│      ↓                                                      │
│  问题：                                                     │
│  1. 库存服务挂了 → 下单失败                                │
│  2. 物流服务响应慢 → 下单慢                                │
│  3. 积分服务升级 → 订单服务也要升级                        │
│  4. 任何一个服务失败 → 下单失败                            │
│  5. 代码复杂，难以维护                                    │
│                                                             │
│  用户感受：                                                 │
│  "为什么下单这么慢？"                                      │
│  "为什么下单失败了？"                                      │
│  "我的订单呢？"                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**消息队列的价值**：
1. **解耦服务**：订单服务不需要知道其他服务存在
2. **异步处理**：下单立即返回，其他操作异步执行
3. **削峰填谷**：应对流量高峰
4. **提高可靠性**：消息持久化，不丢失
5. **扩展性强**：新增服务只需订阅消息

## 第一步：理解消息队列

### 什么是消息队列？

```
┌─────────────────────────────────────────────────────────────┐
│              消息队列（Message Queue）                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [生产者] ── 发送消息 ──→ [消息队列] ←── 消费消息 ── [消费者]│
│                                         ↑                   │
│                                         │                   │
│                                    [消费者2]               │
│                                         ↑                   │
│                                         │                   │
│                                    [消费者3]               │
│                                                             │
│  核心概念：                                                 │
│  - 生产者（Producer）：发送消息的服务                       │
│  - 消息（Message）：要传递的数据                            │
│  - 队列（Queue）：存储消息的容器                            │
│  - 消费者（Consumer）：处理消息的服务                       │
│  - 交换机（Exchange）：路由消息到队列                       │
│                                                             │
│  常见实现：                                                 │
│  - RabbitMQ：功能强大，协议支持多                          │
│  - RocketMQ：阿里开源，高性能                              │
│  - Kafka：高吞吐，适合大数据                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 消息队列模式

```
┌─────────────────────────────────────────────────────────────┐
│  1. 点对点模式（Point-to-Point）                            │
│                                                             │
│  [生产者1] ──→ [队列1] ←── [消费者1]                       │
│  [生产者2] ──→        ←── [消费者2]                       │
│                                                             │
│  特点：一条消息只被一个消费者消费                           │
│  场景：订单处理                                            │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  2. 发布订阅模式（Publish-Subscribe）                        │
│                                                             │
│            [交换机]                                        │
│               ↑                                            │
│  [生产者] ──→     │                                        │
│               │    ├──→ [队列1] ←── [消费者1]              │
│               │    ├──→ [队列2] ←── [消费者2]              │
│               │    └──→ [队列3] ←── [消费者3]              │
│                                                             │
│  特点：一条消息被多个消费者消费                             │
│  场景：广播通知                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 第三步：订单解耦实战

### 场景分析

```
订单服务需要通知：
1. 库存服务：扣减库存
2. 物流服务：创建运单
3. 积分服务：增加积分
4. 通知服务：发送短信
5. 数据服务：记录日志

没有MQ：
订单服务依次调用各服务
→ 耦合严重
→ 任何一个服务失败 → 下单失败
→ 响应慢

有MQ：
订单服务发送消息到MQ
→ 各服务订阅消息
→ 异步处理
→ 解耦
```

#### 2. 配置队列和交换机

```java
// config/RabbitMQConfig.java
package com.taoshu.config;

import org.springframework.amqp.core.*;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class RabbitMQConfig {

    # 交换机（主题模式）
    @Bean
    public TopicExchange orderExchange() {
        return new TopicExchange("order.exchange", true, false);
    }

    # 队列1：库存服务
    @Bean
    public Queue inventoryQueue() {
        return QueueBuilder.durable("order.inventory.queue").build();
    }

    # 队列2：物流服务
    @Bean
    public Queue logisticsQueue() {
        return QueueBuilder.durable("order.logistics.queue").build();
    }

    # 队列3：积分服务
    @Bean
    public Queue pointsQueue() {
        return QueueBuilder.durable("order.points.queue").build();
    }

    # 队列4：通知服务
    @Bean
    public Queue notificationQueue() {
        return QueueBuilder.durable("order.notification.queue").build();
    }

    # 绑定：inventory.* 路由到库存队列
    @Bean
    public Binding inventoryBinding() {
        return BindingBuilder.bind(inventoryQueue())
            .to(orderExchange())
            .with("order.created.inventory");
    }

    # 绑定：logistics.* 路由到物流队列
    @Bean
    public Binding logisticsBinding() {
        return BindingBuilder.bind(logisticsQueue())
            .to(orderExchange())
            .with("order.created.logistics");
    }

    # 绑定：points.* 路由到积分队列
    @Bean
    public Binding pointsBinding() {
        return BindingBuilder.bind(pointsQueue())
            .to(orderExchange())
            .with("order.created.points");
    }

    # 绑定：notification.* 路由到通知队列
    @Bean
    public Binding notificationBinding() {
        return BindingBuilder.bind(notificationQueue())
            .to(orderExchange())
            .with("order.created.notification");
    }
}
```

**路由键设计**

```
order.created.inventory → 库存服务
order.created.logistics → 物流服务
order.created.points → 积分服务
order.created.notification → 通知服务

使用通配符：
order.created.* → 匹配所有
order.created.* → 匹配所有创建事件
order.* → 匹配所有订单事件
```

#### 3. 发送消息

```java
@Service
public class OrderService {

    @Autowired
    private RabbitTemplate rabbitTemplate;

    @Transactional(rollbackFor = Exception.class)
    public Order createOrder(CreateOrderRequest request) {
        # 1. 创建订单
        Order order = buildOrder(request);
        orderRepository.save(order);

        # 2. 扣减库存（同步，保证数据一致性）
        decreaseStock(order);

        # 3. 发送消息到MQ（异步）
        OrderMessage message = new OrderMessage();
        message.setOrderId(order.getId());
        message.setOrderNo(order.getOrderNo());
        message.setUserId(order.getUserId());
        message.setSellerId(order.getSellerId());
        message.setBookId(order.getBookId());
        message.setQuantity(order.getQuantity());
        message.setTotalAmount(order.getTotalAmount());
        message.setCreatedAt(LocalDateTime.now());

        # 发送到不同服务
        rabbitTemplate.convertAndSend(
            "order.exchange",
            "order.created.inventory",
            message
        );

        rabbitTemplate.convertAndSend(
            "order.exchange",
            "order.created.logistics",
            message
        );

        rabbitTemplate.convertAndSend(
            "order.exchange",
            "order.created.points",
            message
        );

        rabbitTemplate.convertAndSend(
            "order.exchange",
            "order.created.notification",
            message
        );

        log.info("订单创建成功，发送消息：{}", order.getOrderNo());

        return order;
    }
}
```

#### 4. 消费消息（库存服务）

```java
// consumer/InventoryConsumer.java
package com.taoshu.consumer;

import com.taoshu.message.OrderMessage;
import com.taoshu.service.InventoryService;
import lombok.extern.slf4j.Slf4j;
import org.springframework.amqp.rabbit.annotation.RabbitListener;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Component;
import com.rabbitmq.client.Channel;
import org.springframework.amqp.core.Message;

@Slf4j
@Component
public class InventoryConsumer {

    @Autowired
    private InventoryService inventoryService;

    /**
     * 消费库存消息
     */
    @RabbitListener(queues = "order.inventory.queue")
    public void handleOrderCreated(
            OrderMessage message,
            Channel channel,
            Message amqpMessage) {

        try {
            log.info("收到库存消息：{}", message);

            # 1. 验证库存
            boolean hasStock = inventoryService.checkStock(
                message.getBookId(),
                message.getQuantity()
            );

            if (!hasStock) {
                log.error("库存不足：orderId={}, bookId={}",
                    message.getOrderId(), message.getBookId());
                # 拒绝消息，不重新入队
                channel.basicNack(amqpMessage.getMessageProperties().getDeliveryTag(), false, false);
                return;
            }

            # 2. 预扣库存（预留）
            inventoryService.decreaseStock(
                message.getBookId(),
                message.getQuantity()
            );

            # 3. 手动确认消息
            channel.basicAck(amqpMessage.getMessageProperties().getDeliveryTag(), false);

            log.info("库存处理完成：orderId={}", message.getOrderId());

        } catch (Exception e) {
            log.error("处理库存消息失败", e);
            try {
                # 拒绝消息，重新入队
                channel.basicNack(amqpMessage.getMessageProperties().getDeliveryTag(), false, true);
            } catch (Exception ex) {
                log.error("拒绝消息失败", ex);
            }
        }
    }
}
```

## 第四步：削峰填谷

### 问题：秒杀场景如何应对？

```
秒杀场景：
10000人同时在1秒内下单
→ 10000个请求同时打到数据库
→ 数据库连接池耗尽
→ 响应变慢
→ 数据库崩溃

解决方案：消息队列削峰
```

### 代码实现：削峰

```java
// service/SeckillService.java
@Service
public class SeckillService {

    @Autowired
    private RabbitTemplate rabbitTemplate;

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    /**
     * 秒杀下单（快速响应）
     */
    @Transactional(rollbackFor = Exception.class)
    public String seckill(Long userId, Long bookId) {
        # 1. Redis预减库存
        String key = "seckill:stock:" + bookId;
        Long stock = redisTemplate.opsForValue().decrement(key);

        if (stock < 0) {
            # 库存不足，恢复
            redisTemplate.opsForValue().increment(key);
            throw new BusinessException("库存不足");
        }

        # 2. 创建订单（简化）
        Order order = new Order();
        order.setUserId(userId);
        order.setBookId(bookId);
        order.setOrderNo(generateOrderNo());
        order.setStatus(0);  # 待处理
        orderRepository.save(order);

        # 3. 发送消息到队列（异步处理）
        SeckillMessage message = new SeckillMessage();
        message.setOrderId(order.getId());
        message.setUserId(userId);
        message.setBookId(bookId);

        rabbitTemplate.convertAndSend("seckill.queue", message);

        # 4. 立即返回
        return "下单成功，请等待处理";
    }
}
```

```java
// consumer/SeckillConsumer.java
/**
 * 秒杀消费者（限流消费）
 */
@Slf4j
@Component
public class SeckillConsumer {

    @Autowired
    private OrderService orderService;

    /**
     * 消费秒杀消息（限制消费速率）
     */

        queues = "seckill.queue",

    )
    public void handleSeckill(SeckillMessage message) {
        try {
            log.info("处理秒杀订单：{}", message.getOrderId());

            # 处理订单（扣库存、创建订单等）
            orderService.processSeckillOrder(message);

            log.info("秒杀订单处理完成：{}", message.getOrderId());

        } catch (Exception e) {
            log.error("处理秒杀订单失败", e);
        }
    }
}
```

```java
// config/RabbitMQConfig.java
/**
 * 秒杀消费者工厂（限流配置）
 */
@Bean
public RabbitListenerContainerFactory<SimpleMessageListenerContainer> seckillListenerFactory(
        ConnectionFactory connectionFactory) {

    SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);

    # 设置并发消费者数量
    factory.setConcurrentConsumers(5);   # 最小5个
    factory.setMaxConcurrentConsumers(20); # 最大20个

    # 设置预取数量（每个消费者每次拉取的消息数）
    factory.setPrefetch(1);

    return factory;
}
```

**削峰效果**

```
没有削峰：
1秒内10000个请求 → 数据库 → 崩溃

有削峰：
1秒内10000个请求 → MQ队列（积压） → 消费者慢慢处理（20个并发）
→ 数据库压力可控
→ 请求成功

消费者配置：
- 5-20个并发消费者
- 每个消费者每次处理1条消息
- 每秒处理：20 × 1 = 20个订单
→ 10000个订单需要：10000 / 20 = 500秒
→ 数据库压力：每秒20个查询（正常负载）
```

## 第五步：消息可靠性

### 问题：如何保证消息不丢失？

```
消息丢失场景：
1. 生产者发送消息失败
2. 消息到达MQ，MQ宕机，消息未持久化
3. 消费者接收消息，处理失败，消息已从队列删除
```

### 可靠性方案

```
┌─────────────────────────────────────────────────────────────┐
│              消息可靠性保障                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [生产者]                                                    │
│      ↓                                                      │
│  1. 发送确认（Publisher Confirm）                            │
│     - 确保消息到达MQ                                        │
│      ↓                                                      │
│  [RabbitMQ]                                                 │
│      ↓                                                      │
│  2. 消息持久化（Durable）                                    │
│     - MQ重启，消息不丢失                                    │
│      ↓                                                      │
│  3. 队列持久化                                               │
│     - Queue durable = true                                  │
│      ↓                                                      │
│  [消费者]                                                    │
│      ↓                                                      │
│  4. 手动确认（Manual Ack）                                  │
│     - 处理成功才确认                                        │
│     - 失败重新入队                                          │
│      ↓                                                      │
│  完成                                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 代码实现：可靠消息

```java
// config/RabbitMQConfig.java
@Bean
public ConnectionFactory connectionFactory() {
    CachingConnectionFactory connectionFactory = new CachingConnectionFactory("localhost");
    connectionFactory.setUsername("guest");
    connectionFactory.setPassword("guest");

    # 开启发送确认
    connectionFactory.setPublisherConfirmType(CachingConnectionFactory.ConfirmType.CORRELATED);

    # 开启返回确认
    connectionFactory.setPublisherReturns(true);

    return connectionFactory;
}

@Bean
public RabbitTemplate rabbitTemplate(ConnectionFactory connectionFactory) {
    RabbitTemplate rabbitTemplate = new RabbitTemplate(connectionFactory);

    # 发送确认回调
    rabbitTemplate.setConfirmCallback((correlationData, ack, cause) -> {
        if (ack) {
            log.info("消息发送成功");
        } else {
            log.error("消息发送失败：{}", cause);
            # 重试或记录到失败队列
        }
    });

    # 返回确认回调（消息未路由到队列）
    rabbitTemplate.setReturnsCallback(returned -> {
        log.error("消息未路由到队列：{}", returned.getMessage());
        # 记录到失败队列
    });

    return rabbitTemplate;
}
```

```java
// 消费者手动确认
@RabbitListener(queues = "order.queue")
public void handleOrder(
        OrderMessage message,
        Channel channel,
        Message amqpMessage) throws IOException {

    try {
        # 处理业务逻辑
        processOrder(message);

        # 手动确认
        channel.basicAck(
            amqpMessage.getMessageProperties().getDeliveryTag(),
            false  # 不批量确认
        );

    } catch (Exception e) {
        log.error("处理消息失败", e);

        # 拒绝消息，重新入队
        channel.basicNack(
            amqpMessage.getMessageProperties().getDeliveryTag(),
            false,  # 不批量
            true    # 重新入队
        );
    }
}
```

## 第六步：幂等性处理

### 问题：消息重复消费怎么办？

```
场景：
1. 消费者处理消息
2. 发送ACK确认
3. 网络中断，ACK未到达
4. MQ重新投递消息
5. 消费者重复处理

后果：
- 订单重复创建
- 库存重复扣减
- 积分重复增加
```

### 解决方案：幂等性

```java
// service/OrderService.java
@Service
public class OrderService {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    private static final String PROCESSED_ORDER_KEY = "order:processed:";

    /**
     * 处理订单（幂等）
     */
    public void processOrder(OrderMessage message) {
        # 1. 检查是否已处理
        String key = PROCESSED_ORDER_KEY + message.getOrderNo();
        Boolean processed = redisTemplate.hasKey(key);

        if (Boolean.TRUE.equals(processed)) {
            log.info("订单已处理，跳过：{}", message.getOrderNo());
            return;  # 已处理，直接返回
        }

        # 2. 处理订单
        createOrder(message);

        # 3. 标记已处理（7天过期）
        redisTemplate.opsForValue().set(key, "1", 7, TimeUnit.DAYS);

        log.info("订单处理完成：{}", message.getOrderNo());
    }

    /**
     * 基于数据库唯一键的幂等
     */
    @Transactional(rollbackFor = Exception.class)
    public void createOrder(OrderMessage message) {
        try {
            Order order = new Order();
            order.setOrderNo(message.getOrderNo());
            # ... 其他字段
            orderRepository.save(order);

        } catch (DuplicateKeyException e) {
            # 订单号唯一，重复插入会报错
            log.info("订单已存在：{}", message.getOrderNo());
        }
    }
}
```

**幂等性方案对比**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Redis去重 | 快速 | 重启丢失 | 短期去重 |
| 数据库唯一键 | 持久化 | 性能差 | 订单类 |
| 状态机幂等 | 逻辑严谨 | 复杂 | 状态变更 |
| Token机制 | 通用 | 需要预生成 | 通用场景 |

### 3. 死信队列

```java
# 消息处理失败后，进入死信队列
@Bean
public Queue deadLetterQueue() {
    return QueueBuilder.durable("order.dead.queue").build();
}

@Bean
public DirectExchange deadLetterExchange() {
    return new DirectExchange("order.dead.exchange");
}

@Bean
public Binding deadLetterBinding() {
    return BindingBuilder.bind(deadLetterQueue())
        .to(deadLetterExchange())
        .with("order.dead");
}
```
