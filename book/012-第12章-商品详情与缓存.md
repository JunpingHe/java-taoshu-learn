# 第十二章：商品详情与缓存

> **小王的疑问**：用户点击一本书查看详情，要加载书名、作者、价格、卖家信息、评论...每次都查数据库？10万用户同时访问，数据库会不会崩溃？

---

## 🤔 为什么需要缓存？

### 真实场景：数据库崩溃事件

```
某电商平台黑色星期五：
┌─────────────────────────────────────────────────────────────┐
│  00:00  秒杀开始                                            │
│  00:01  流量峰值：10万QPS（每秒10万请求）                   │
│  00:02  数据库CPU：100%                                     │
│  00:03  响应时间：从50ms → 5000ms                          │
│  00:04  数据库连接池：耗尽                                  │
│  00:05  数据库：崩溃！                                      │
│                                                             │
│  结果：                                                     │
│  - 用户看到"系统繁忙"                                      │
│  - 订单无法创建                                            │
│  - 损失：销售额减少80%                                      │
└─────────────────────────────────────────────────────────────┘
```

**问题根源**：
```
用户访问商品详情 → 查数据库
  ↓
商品详情数据不变（书名、作者、ISBN...）
  ↓
但每次都查数据库！
  ↓
10万用户访问 = 10万次查询 = 数据库崩溃
```

### ✅ 使用缓存的解决方案

```
┌─────────────────────────────────────────────────────────────┐
│                    缓存架构图                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户请求商品详情                                           │
│      ↓                                                      │
│  先查Redis缓存（内存，超快！）                               │
│      ↓                                                      │
│  ├─ 缓存命中？                                             │
│  │   ├─ 是 → 直接返回（0.1ms）                             │
│  │   └─ 否 → 查数据库 → 写入缓存 → 返回                    │
│                                                             │
│  对比：                                                     │
│  MySQL查询：50ms（磁盘IO）                                  │
│  Redis查询：0.1ms（内存）                                   │
│  速度提升：500倍！                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**缓存的价值**：
1. **性能提升**：响应时间从50ms → 0.1ms
2. **保护数据库**：减少90%以上的数据库查询
3. **降低成本**：同样配置的机器，能承受更多流量
4. **提升体验**：用户秒开页面，不流失

#### 3. 在Service中实现查询逻辑

```java
// service/BookService.java
@Service
public class BookService {

    @Autowired
    private BookRepository bookRepository;

    @Autowired
    private UserRepository userRepository;

    @Autowired
    private BookImageRepository imageRepository;

    # 查询商品详情
    public BookDetailResponse getBookDetail(Long bookId) {
        # 1. 查询商品
        Book book = bookRepository.findDetailById(bookId)
            .orElseThrow(() -> new BusinessException("商品不存在"));

        # 2. 查询卖家信息
        User seller = userRepository.findById(book.getSellerId())
            .orElseThrow(() -> new BusinessException("卖家信息异常"));

        # 3. 查询商品图片
        List<BookImage> images = imageRepository.findByBookIdOrderBySortOrder(bookId);

        # 4. 增加浏览次数
        incrementViewCount(bookId);

        # 5. 组装响应
        return BookDetailResponse.fromEntity(book, seller, images);
    }

    # 增加浏览次数（异步）
    @Async
    public void incrementViewCount(Long bookId) {
        bookRepository.incrementViewCount(bookId);
    }
}
```

**🤔 为什么浏览次数异步更新？**

```
同步方式（慢）：
用户请求 → 查数据库 → 更新浏览次数 → 返回
耗时：50ms + 20ms = 70ms

异步方式（快）：
用户请求 → 查数据库 → 立即返回
              → 异步更新浏览次数
耗时：50ms（用户无感知）

原理：浏览次数更新不要求实时，允许短暂延迟
```

## 第二步：使用Spring Cache

### 问题：如何优雅地使用缓存？

#### ❌ 糟糕做法：手动操作Redis

```java
// ❌ 糟糕的代码：手动缓存
public BookDetailResponse getBookDetail(Long bookId) {
    # 1. 手动查缓存
    String key = "book:detail:" + bookId;
    String cached = redisTemplate.opsForValue().get(key);

    if (cached != null) {
        # 2. 缓存命中，反序列化
        return JSON.parseObject(cached, BookDetailResponse.class);
    }

    # 3. 缓存未命中，查数据库
    BookDetailResponse detail = buildFromDatabase(bookId);

    # 4. 写入缓存
    redisTemplate.opsForValue().set(key, JSON.toJSONString(detail), 30, TimeUnit.MINUTES);

    return detail;
}

# 问题：
# 1. 代码冗长，每个方法都要写一遍
# 2. 容易出错：忘记写缓存、忘记更新缓存
# 3. 缓存逻辑和业务逻辑混在一起
```

#### ✅ 正确做法：Spring Cache注解

```java
// ✅ 优雅的代码：声明式缓存
@Cacheable(value = "book:detail", key = "#bookId", unless = "#result == null")
public BookDetailResponse getBookDetail(Long bookId) {
    # 只需写业务逻辑，缓存自动处理！
    return buildFromDatabase(bookId);
}

# 优点：
# 1. 代码简洁，一个注解搞定
# 2. 不容易出错：Spring自动处理缓存
# 3. 关注点分离：业务逻辑和缓存逻辑分离
```

#### 3. 配置缓存管理器

```java
// config/CacheConfig.java
package com.taoshu.config;

import com.fasterxml.jackson.annotation.JsonAutoDetect;
import com.fasterxml.jackson.annotation.PropertyAccessor;
import com.fasterxml.jackson.databind.ObjectMapper;
import com.fasterxml.jackson.databind.jsontype.impl.LaissezFaireSubTypeValidator;
import org.springframework.cache.CacheManager;
import org.springframework.cache.annotation.EnableCaching;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.data.redis.cache.RedisCacheConfiguration;
import org.springframework.data.redis.cache.RedisCacheManager;
import org.springframework.data.redis.connection.RedisConnectionFactory;
import org.springframework.data.redis.serializer.Jackson2JsonRedisSerializer;
import org.springframework.data.redis.serializer.RedisSerializationContext;
import org.springframework.data.redis.serializer.StringRedisSerializer;

import java.time.Duration;

@Configuration
@EnableCaching  # 启用缓存
public class CacheConfig {

    @Bean
    public CacheManager cacheManager(RedisConnectionFactory factory) {
        # 1. 配置JSON序列化
        Jackson2JsonRedisSerializer<Object> serializer = new Jackson2JsonRedisSerializer<>(Object.class);

        ObjectMapper objectMapper = new ObjectMapper();
        objectMapper.setVisibility(PropertyAccessor.ALL, JsonAutoDetect.Visibility.ANY);
        objectMapper.activateDefaultTyping(
            LaissezFaireSubTypeValidator.instance,
            ObjectMapper.DefaultTyping.NON_FINAL
        );
        serializer.setObjectMapper(objectMapper);

        # 2. 配置缓存规则
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30))  # 默认30分钟过期
            .serializeKeysWith(RedisSerializationContext.SerializationPair.fromSerializer(new StringRedisSerializer()))
            .serializeValuesWith(RedisSerializationContext.SerializationPair.fromSerializer(serializer))
            .disableCachingNullValues();  # 不缓存null值

        # 3. 针对不同缓存设置不同过期时间
        return RedisCacheManager.builder(factory)
            .cacheDefaults(config)
            .withCacheConfiguration("book:detail",  # 商品详情缓存
                config.entryTtl(Duration.ofHours(1)))  # 1小时
            .withCacheConfiguration("category:list",  # 分类列表缓存
                config.entryTtl(Duration.ofDays(1)))   # 1天
            .build();
    }
}
```

**🤔 为什么序列化很重要？**

```
问题：Java对象不能直接存Redis

方案1：JDK序列化（默认）
对象 → 二进制 → Redis
问题：可读性差，占用空间大

方案2：JSON序列化（推荐！）
对象 → JSON字符串 → Redis
优点：可读性好，占用空间小，跨语言
```

**Spring Cache注解详解**

| 注解 | 作用 | 执行时机 |
|------|------|----------|
| `@Cacheable` | 查询时，先查缓存，缓存未命中才查数据库 | 查询方法 |
| `@CachePut` | 无论缓存是否有，都查数据库并更新缓存 | 更新方法 |
| `@CacheEvict` | 删除缓存 | 删除/更新方法 |
| `@Caching` | 组合多个缓存操作 | 复杂场景 |
| `@CacheConfig` | 类级别缓存配置 | 类上 |

## 第三步：缓存更新策略

### 问题：商品更新后，缓存怎么办？

#### 场景分析

```
场景：卖家修改商品价格
时间线：
10:00  用户A查询 → 缓存《Java核心技术》价格=50元
10:05  卖家修改价格 → 数据库价格=40元，但缓存还是50元
10:10  用户B查询 → 缓存命中 → 显示价格=50元（错误！）

问题：缓存和数据库不一致！
```

#### 解决方案

```
方案1：更新时清除缓存（推荐！）
更新数据库 → 清除缓存 → 下次查询重新加载
优点：简单，不会出现不一致
缺点：下次查询需要查数据库

方案2：更新时更新缓存（双写）
更新数据库 → 更新缓存
优点：下次查询更快
缺点：双写可能失败，需要事务保证

方案3：延迟双写（MQ）
更新数据库 → 发消息 → 异步更新缓存
优点：解耦，性能好
缺点：复杂度高

淘书网选择：方案1（更新时清除缓存）
```

### 代码实现：缓存清除

```java
// service/BookService.java
@Service
public class BookService {

    # 更新商品信息
    @Transactional
    @CacheEvict(value = "book:detail", key = "#bookId")
    public void updateBook(Long bookId, BookUpdateRequest request) {
        # 1. 查询商品
        Book book = bookRepository.findById(bookId)
            .orElseThrow(() -> new BusinessException("商品不存在"));

        # 2. 更新字段
        if (request.getPrice() != null) {
            book.setPrice(request.getPrice());
        }
        if (request.getDescription() != null) {
            book.setDescription(request.getDescription());
        }
        # ... 其他字段

        # 3. 保存到数据库
        bookRepository.save(book);

        # 4. 缓存自动清除（@CacheEvict）
    }

    # 批量清除缓存（比如分类下所有商品）
    @CacheEvict(value = "book:detail", allEntries = true)
    public void clearAllBookCache() {
        # 清除所有商品缓存
    }

    # 删除商品
    @Transactional
    @CacheEvict(value = "book:detail", key = "#bookId")
    public void deleteBook(Long bookId) {
        # 软删除
        Book book = bookRepository.findById(bookId)
            .orElseThrow(() -> new BusinessException("商品不存在"));
        book.setIsDeleted(true);
        bookRepository.save(book);

        # 缓存自动清除
    }
}
```

## 第四步：防止缓存穿透

### 问题：缓存穿透是什么？

#### 真实场景

```
恶意攻击：
攻击者不断请求不存在的商品ID
GET /api/book/999999
GET /api/book/999998
GET /api/book/999997
...

缓存流程：
1. 查缓存 → 未命中
2. 查数据库 → 不存在
3. 返回null
4. 不写缓存（因为没有数据）

结果：
每次都要查数据库！
数据库压力骤增 → 崩溃！
```

**🤔 为什么叫"穿透"？**

```
正常流程：请求 → 缓存（命中） → 返回
穿透流程：请求 → 缓存（未命中） → 数据库（未命中） → 返回
              ↓
         大量无效请求直接"穿透"缓存，打到数据库
```

### 解决方案

#### 方案1：缓存空值（推荐！）

```java
// service/BookService.java
@Cacheable(
    value = "book:detail",
    key = "#bookId"
    # 注意：不设置 unless = "#result == null"
    # 允许缓存null值
)
public BookDetailResponse getBookDetail(Long bookId) {
    Optional<Book> book = bookRepository.findById(bookId);

    if (!book.isPresent()) {
        # 返回null，Spring Cache会缓存null
        return null;
    }

    return buildDetail(book.get());
}

# 配置：允许缓存null
@Bean
public CacheManager cacheManager(RedisConnectionFactory factory) {
    RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
        # 允许缓存null值
        .disableCachingNullValues();  # 改为 .enableCachingNullValues()
}
```

**⚠️ 注意：空值缓存时间要短**

```java
# 针对空值设置更短的过期时间
@Bean
public CacheManager cacheManager(RedisConnectionFactory factory) {
    RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
        .entryTtl(Duration.ofMinutes(30));  # 正常数据30分钟

    # 空值缓存5分钟（避免恶意攻击占用缓存）
    return RedisCacheManager.builder(factory)
        .cacheDefaults(config)
        .withCacheConfiguration("book:detail:null",
            config.entryTtl(Duration.ofMinutes(5)))
        .build();
}
```

#### 方案2：布隆过滤器（高级）

```
布隆过滤器原理：
1. 将所有商品ID存入布隆过滤器
2. 查询前先检查布隆过滤器
3. 如果布隆过滤器说"不存在"，直接返回，不查数据库

优点：内存占用小（1亿ID只需100MB）
缺点：有误判率（可能说存在，实际不存在）

Redis布隆过滤器实现：
redisTemplate.opsForBloomFilter().add("book:bloom", bookId);
boolean exists = redisTemplate.opsForBloomFilter().mightContain("book:bloom", bookId);
```

## 第五步：防止缓存雪崩

### 问题：缓存雪崩是什么？

#### 真实场景

```
某电商平台双十一：
┌─────────────────────────────────────────────────────────────┐
│  00:00  秒杀开始                                            │
│  00:00  大量商品缓存同时过期（都是凌晨设置的，30分钟过期）  │
│  00:01  大量请求打到数据库                                  │
│  00:02  数据库CPU：100%                                     │
│  00:03  数据库：崩溃！                                      │
│                                                             │
│  雪崩：大量缓存同时失效 → 请求全部打到数据库                │
└─────────────────────────────────────────────────────────────┘
```

### 解决方案

#### 方案1：过期时间随机化（推荐！）

```java
// service/BookService.java
@Cacheable(
    value = "book:detail",
    key = "#bookId"
)
public BookDetailResponse getBookDetail(Long bookId) {
    # 查询数据...
    return detail;
}

# 配置：在写入缓存时，过期时间加随机值
@Bean
public CacheManager cacheManager(RedisConnectionFactory factory) {
    RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
        # 基础过期时间：30分钟
        .entryTtl(Duration.ofMinutes(30))
        # 自定义缓存写入策略
        .serializeValuesWith(...);

    return RedisCacheManager.builder(factory)
        .cacheDefaults(config)
        # 使用自定义CacheWriter
        .cacheWriter(randomExpireCacheWriter(factory))
        .build();
}

# 自定义CacheWriter：随机过期时间
private RedisCacheWriter randomExpireCacheWriter(RedisConnectionFactory factory) {
    return new DefaultRedisCacheWriter(factory) {
        @Override
        public void put(String name, byte[] key, byte[] value, Duration ttl) {
            # 基础30分钟 + 随机0-10分钟
            Duration randomTtl = ttl.plus(Duration.ofMillis(
                ThreadLocalRandom.current().nextLong(0, 600000)
            ));
            super.put(name, key, value, randomTtl);
        }
    };
}
```

**效果**：
```
原来：所有缓存都是30:00过期
现在：缓存过期时间分散在30:00 - 40:00之间
结果：不会同时失效，避免雪崩
```

#### 方案2：多级缓存

```
架构：
┌─────────────────────────────────────────────────────────────┐
│  L1缓存：本地缓存（Caffeine）                                │
```

## 第六步：防止缓存击穿

### 问题：缓存击穿是什么？

#### 真实场景

```
热门商品：《三体》秒杀
时间线：
09:59  1000人在等待
10:00  秒杀开始
10:00  1000个请求同时查询商品详情
10:00  缓存刚好过期（1小时前设置的）
10:00  1000个请求同时查数据库

击穿：热点数据过期瞬间，大量请求打到数据库
```

**🤔 击穿 vs 雪崩 vs 穿透**

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **缓存穿透** | 查询不存在的数据 | 缓存空值 / 布隆过滤器 |
| **缓存雪崩** | 大量缓存同时失效 | 过期时间随机化 / 多级缓存 |
| **缓存击穿** | 热点数据过期瞬间大量请求 | 永不过期 / 互斥锁 |

### 解决方案

#### 方案1：热点数据永不过期

```java
// service/BookService.java
@Cacheable(
    value = "book:detail:hot",  # 热门商品单独缓存
    key = "#bookId"
)
public BookDetailResponse getBookDetail(Long bookId) {
    # 查询数据...
    return detail;
}

# 配置：热门商品缓存不过期
@Bean
public CacheManager cacheManager(RedisConnectionFactory factory) {
    RedisCacheConfiguration hotConfig = RedisCacheConfiguration.defaultCacheConfig()
        .entryTtl(Duration.ofDays(7));  # 7天过期（相当于永不过期）

    return RedisCacheManager.builder(factory)
        .withCacheConfiguration("book:detail:hot", hotConfig)
        .build();
}

# 如何判断热门商品？
# 1. 浏览次数 > 1000
# 2. 销量 > 100
# 3. 手动设置（运营后台）
```

#### 方案2：互斥锁（分布式锁）

```java
// service/BookService.java
public BookDetailResponse getBookDetail(Long bookId) {
    # 1. 查缓存
    String cacheKey = "book:detail:" + bookId;
    BookDetailResponse cached = getFromCache(cacheKey);
    if (cached != null) {
        return cached;
    }

    # 2. 缓存未命中，尝试获取锁
    String lockKey = "lock:book:" + bookId;
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);

    if (Boolean.TRUE.equals(locked)) {
        try {
            # 3. 获得锁，再次检查缓存（双重检查）
            cached = getFromCache(cacheKey);
            if (cached != null) {
                return cached;
            }

            # 4. 查数据库
            BookDetailResponse detail = buildFromDatabase(bookId);

            # 5. 写入缓存
            setCache(cacheKey, detail);

            return detail;
        } finally {
            # 6. 释放锁
            redisTemplate.delete(lockKey);
        }
    } else {
        # 7. 未获得锁，等待100ms后查缓存
        Thread.sleep(100);
        return getFromCache(cacheKey);
    }
}
```

**互斥锁流程图**

```
1000个请求同时到达
    ↓
请求1：获取锁 → 成功 → 查数据库 → 写缓存 → 释放锁
请求2：获取锁 → 失败 → 等待 → 查缓存（已有） → 返回
请求3：获取锁 → 失败 → 等待 → 查缓存（已有） → 返回
...
请求1000：获取锁 → 失败 → 等待 → 查缓存（已有） → 返回

结果：只有1个请求查数据库，999个请求查缓存
```

## ⚠️ 缓存最佳实践

### 1. 缓存Key设计

```java
# ❌ 不好：太简单
@Cacheable(value = "book", key = "#id")

# ✅ 好：包含业务前缀
@Cacheable(value = "book:detail", key = "#id")

# ✅ 更好：包含版本号（便于清除）
@Cacheable(value = "book:detail:v1", key = "#id")

# ✅ 组合Key：多参数
@Cacheable(value = "book:list", key = "#categoryId + ':' + #page + ':' + #size")
```

### 2. 缓存过期时间

```java
# 不同数据不同过期时间
- 商品详情：1小时（更新频率低）
- 商品列表：30分钟（有新商品）
- 热门商品：7天（基本不变）
- 分类列表：1天（很少变化）
- 用户信息：30分钟（可能更新）
```

### 3. 缓存粒度

```
粗粒度缓存：
key = "book:list"
value = 所有商品列表
问题：任何一个商品变化，都要清除整个缓存

细粒度缓存（推荐！）：
key = "book:detail:1"
value = 商品1详情
key = "book:detail:2"
value = 商品2详情
优点：单个商品变化，只清除对应缓存
```

### 4. 监控告警

```java
# 监控指标
- 缓存命中率：>90%为健康
- 缓存响应时间：<1ms
- 缓存大小：避免内存溢出
- 缓存穿透数：告警阈值

# 实现监控
@Component
public class CacheMonitor {
    @EventListener
    public void onCacheHit(CacheHitEvent event) {
        # 记录命中
        metricsService.increment("cache.hit");
    }

    @EventListener
    public void onCacheMiss(CacheMissEvent event) {
        # 记录未命中
        metricsService.increment("cache.miss");
    }
}
```

## 📊 Django vs Spring Boot 对比

| 功能 | Django | Spring Boot | 难度 |
|------|--------|-------------|------|
| 缓存框架 | `django.core.cache` | `Spring Cache` | 相当 |
| Redis集成 | `django-redis` | `spring-data-redis` | 相当 |
| 注解缓存 | 无装饰器 | `@Cacheable` 等注解 | Spring Boot更优雅 |
| 缓存更新 | 手动 | `@CacheEvict` | Spring Boot更方便 |
| 多级缓存 | 手动 | 自动 | Spring Boot更强 |

**Django优势**：
- 配置简单，开箱即用

**Spring Boot优势**：
- 注解驱动，代码更优雅
- 支持多种缓存提供商（Redis、Caffeine、Ehcache）
- 支持缓存抽象，便于切换

## 🎯 本章小结

| 知识点 | 为什么需要 | 怎么做 |
|--------|-----------|--------|
| **缓存** | 减少数据库查询，提升性能 | Redis + Spring Cache |
| **@Cacheable** | 查询时自动使用缓存 | 注解方法，自动缓存返回值 |
| **@CacheEvict** | 更新时清除缓存，避免不一致 | 注解更新方法 |
| **缓存穿透** | 恶意查询不存在的数据 | 缓存空值 / 布隆过滤器 |
| **缓存雪崩** | 大量缓存同时失效 | 过期时间随机化 / 多级缓存 |
| **缓存击穿** | 热点数据过期瞬间大量请求 | 永不过期 / 分布式锁 |

**记忆口诀**：
- **查数据用@Cacheable**，**更新数据用@CacheEvict**
- **穿透缓存空值**，**雪崩随机过期**，**击穿加锁防**
- **热点永不过期**，**多级缓存更稳**
