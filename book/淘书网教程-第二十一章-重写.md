## 🤔 为什么需要日志与监控？

### 真实场景：没有日志的痛苦

```
某电商平台没有完善的日志和监控：
┌─────────────────────────────────────────────────────────────┐
│  用户反馈："我下单失败了！"                                 │
│                                                             │
│  开发人员排查：                                             │
│  1. 登录服务器 → 查看应用日志                              │
│  2. 日志文件太大（10GB）→ 打开卡死                         │
│  3. 日志没有分类 → 找不到关键信息                          │
│  4. 日志没有上下文 → 不知道用户是谁                        │
│  5. 日志没有堆栈 → 不知道错误原因                          │
│  6. 排查2小时 → 原来是数据库连接池耗尽                      │
│                                                             │
│  问题：                                                     │
│  - 排查困难                                                │
│  - 耗时很长                                                │
│  - 用户体验差                                              │
│  - 同类问题重复发生                                        │
│                                                             │
│  有完善的日志和监控：                                       │
│  1. 日志聚合（ELK）→ 快速搜索                              │
│  2. 日志分类（INFO/WARN/ERROR）→ 过滤查看                  │
│  3. 日志上下文（traceId）→ 追踪请求链路                    │
│  4. 实时监控（Grafana）→ 发现异常立即告警                  │
│  5. 排查时间：2分钟                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**日志与监控的价值**：
1. **快速定位问题**：从几小时缩短到几分钟
2. **主动发现**：监控告警，用户反馈前先处理
3. **数据分析**：了解系统运行状况
4. **性能优化**：发现瓶颈
5. **安全审计**：追踪操作记录

## 第一步：日志框架

### 问题：如何正确记录日志？

#### ❌ 错误做法

```java
// 错误1：使用System.out.println
System.out.println("订单创建成功");
# 问题：
# - 无法控制日志级别
# - 无法输出到文件
# - 性能差（同步输出）

// 错误2：日志级别混乱
log.info("用户不存在：{}", userId);  # 应该用error
log.error("订单创建成功");            # 应该用info

// 错误3：缺少关键信息
log.info("订单创建成功");  # 不知道哪个订单、哪个用户

// 错误4：大量调试日志
log.debug("变量x的值：{}", x);  # 生产环境输出大量无用日志

// 错误5：异常信息丢失
try {
    # 业务逻辑
} catch (Exception e) {
    log.error("处理失败");  # 丢失异常堆栈
}
```

#### ✅ 正确做法

```java
// 正确1：使用SLF4J
import lombok.extern.slf4j.Slf4j;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Slf4j  # Lombok自动生成log变量
public class OrderService {

    # 或者手动创建
    // private static final Logger log = LoggerFactory.getLogger(OrderService.class);

    public void createOrder() {
        log.info("开始创建订单，userId：{}", userId);
    }
}

// 正确2：正确的日志级别
log.trace("最详细的日志")     # 追踪问题
log.debug("调试信息")          # 开发调试
log.info("关键业务流程")      # 业务流程
log.warn("警告信息")          # 潜在问题
log.error("错误信息")         # 错误，需要处理

// 正确3：包含关键信息
log.info("订单创建成功，orderNo：{}, userId：{}, amount：{}",
    orderNo, userId, amount);

// 正确4：正确的异常处理
try {
    # 业务逻辑
} catch (Exception e) {
    log.error("订单创建失败，userId：{}", userId, e);
    # 第三个参数传入异常，会输出完整堆栈
}

// 正确5：使用占位符（性能好）
log.info("订单：{}，用户：{}", orderNo, userId);
# 而不是：
log.info("订单：" + orderNo + "，用户：" + userId);  # 字符串拼接，性能差
```

## 第二步：日志配置

### 问题：如何配置日志？

#### logback.xml配置

```xml
<!-- src/main/resources/logback.xml -->
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <!-- 日志输出格式 -->
    <property name="LOG_PATTERN"
              value="%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} [%X{traceId}] - %msg%n"/>

    <!-- 日志文件路径 -->
    <property name="LOG_PATH" value="logs"/>

    <!-- 控制台输出 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>${LOG_PATTERN}</pattern>
            <charset>UTF-8</charset>
        </encoder>
    </appender>

    <!-- 文件输出：所有日志 -->
    <appender name="FILE_ALL" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>${LOG_PATH}/taoshu-all.log</file>
        <encoder>
            <pattern>${LOG_PATTERN}</pattern>
            <charset>UTF-8</charset>
        </encoder>
        <!-- 滚动策略 -->
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>${LOG_PATH}/taoshu-all.%d{yyyy-MM-dd}.%i.log</fileNamePattern>
            <maxHistory>30</maxHistory>  <!-- 保留30天 -->
            <timeBasedFileNamingAndTriggeringPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
                <maxFileSize>100MB</maxFileSize>  <!-- 单文件最大100MB -->
            </timeBasedFileNamingAndTriggeringPolicy>
        </rollingPolicy>
    </appender>

    <!-- 文件输出：ERROR日志 -->
    <appender name="FILE_ERROR" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>${LOG_PATH}/taoshu-error.log</file>
        <encoder>
            <pattern>${LOG_PATTERN}</pattern>
            <charset>UTF-8</charset>
        </encoder>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>${LOG_PATH}/taoshu-error.%d{yyyy-MM-dd}.%i.log</fileNamePattern>
            <maxHistory>30</maxHistory>
            <timeBasedFileNamingAndTriggeringPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
                <maxFileSize>50MB</maxFileSize>
            </timeBasedFileNamingAndTriggeringPolicy>
        </rollingPolicy>
        <!-- 只记录ERROR级别 -->
        <filter class="ch.qos.logback.classic.filter.LevelFilter">
            <level>ERROR</level>
            <onMatch>ACCEPT</onMatch>
            <onMismatch>DENY</onMismatch>
        </filter>
    </appender>

    <!-- 异步输出：性能优化 -->
    <appender name="ASYNC_FILE" class="ch.qos.logback.classic.AsyncAppender">
        <appender-ref ref="FILE_ALL"/>
        <queueSize>512</queueSize>  <!-- 队列大小 -->
        <discardingThreshold>0</discardingThreshold>  <!-- 不丢弃任何日志 -->
    </appender>

    <!-- 开发环境：输出到控制台 -->
    <springProfile name="dev">
        <root level="DEBUG">
            <appender-ref ref="CONSOLE"/>
        </root>
    </springProfile>

    <!-- 生产环境：输出到文件 -->
    <springProfile name="prod">
        <root level="INFO">
            <appender-ref ref="CONSOLE"/>
            <appender-ref ref="ASYNC_FILE"/>
            <appender-ref ref="FILE_ERROR"/>
        </root>
    </springProfile>

    <!-- 特定包的日志级别 -->
    <logger name="com.taoshu" level="DEBUG"/>
    <logger name="org.springframework" level="INFO"/>
    <logger name="org.hibernate.SQL" level="DEBUG"/>  <!-- 显示SQL -->
    <logger name="org.hibernate.type.descriptor.sql.BasicBinder" level="TRACE"/>  <!-- 显示参数值 -->
</configuration>
```

## 第三步：请求链路追踪

### 问题：如何追踪一个请求的完整链路？

```
场景：
用户下单 → 创建订单 → 扣减库存 → 发送通知
      → 某个环节失败了

问题：
1. 每个服务的日志独立
2. 无法关联是哪个用户的请求
3. 排查困难

解决：
使用traceId追踪请求链路
```

### 代码实现：MDC追踪

```java
// filter/TraceIdFilter.java
package com.taoshu.filter;

import jakarta.servlet.*;
import org.slf4j.MDC;
import org.springframework.stereotype.Component;
import java.io.IOException;
import java.util.UUID;

@Component
public class TraceIdFilter implements Filter {

    private static final String TRACE_ID = "traceId";

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {

        # 1. 生成traceId
        String traceId = UUID.randomUUID().toString().replace("-", "");

        # 2. 放入MDC
        MDC.put(TRACE_ID, traceId);

        try {
            # 3. 继续执行
            chain.doFilter(request, response);

        } finally {
            # 4. 清除MDC
            MDC.remove(TRACE_ID);
        }
    }
}
```

**日志输出示例**

```
2023-12-01 10:30:15.123 [http-nio-8080-exec-1] INFO  c.t.service.OrderService [a1b2c3d4e5f6g7h8] - 订单创建成功，orderNo：TS20231201103015123
2023-12-01 10:30:15.124 [http-nio-8080-exec-1] INFO  c.t.service.InventoryService [a1b2c3d4e5f6g7h8] - 库存扣减成功，bookId：123
2023-12-01 10:30:15.500 [http-nio-8080-exec-1] INFO  c.t.service.NotificationService [a1b2c3d4e5f6g7h8] - 短信发送成功

同一请求的日志有相同的traceId，可以快速关联
```

## 第四步：结构化日志

### 问题：如何让日志更易于查询分析？

```
普通日志：
2023-12-01 10:30:15 INFO 订单创建成功，orderNo：TS123，userId：456

问题：
- 非结构化，难以解析
- 无法按字段查询

结构化日志：
{"timestamp": "2023-12-01T10:30:15.123", "level": "INFO", "logger": "OrderService", "traceId": "abc123", "message": "订单创建成功", "orderNo": "TS123", "userId": 456}

优点：
- JSON格式，易于解析
- 可按任意字段查询
- 支持日志聚合分析
```

### 代码实现：结构化日志

```xml
<!-- logback-spring.xml -->
<appender name="JSON_FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>${LOG_PATH}/taoshu-json.log</file>
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
        <!-- 包含MDC -->
        <includeMdcKeyName>traceId</includeMdcKeyName>
        <includeMdcKeyName>userId</includeMdcKeyName>
        <!-- 自定义字段 -->
        <fieldNames>
            <timestamp>timestamp</timestamp>
            <level>level</level>
            <logger>logger</logger>
            <message>message</message>
        </fieldNames>
    </encoder>
    <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
        <fileNamePattern>${LOG_PATH}/taoshu-json.%d{yyyy-MM-dd}.log</fileNamePattern>
        <maxHistory>30</maxHistory>
    </rollingPolicy>
</appender>
```

**使用LogstashEncoder**

```xml
<!-- pom.xml -->
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

### 问题：如何监控应用健康状态？

```
监控指标：
1. 应用是否运行
2. JVM内存使用情况
3. 线程池状态
```

### 代码实现：Actuator配置

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.springframework.boot</groupId>
```

### 自定义健康检查

```java
// health/CustomHealthIndicator.java
package com.taoshu.health;

import org.springframework.boot.actuate.health.Health;
import org.springframework.boot.actuate.health.HealthIndicator;
import org.springframework.stereotype.Component;
import java.lang.management.ManagementFactory;
import java.lang.management.MemoryMXBean;

@Component
public class CustomHealthIndicator implements HealthIndicator {

    @Override
    public Health health() {
        # 1. 检查内存使用
        MemoryMXBean memoryBean = ManagementFactory.getMemoryMXBean();
        long used = memoryBean.getHeapMemoryUsage().getUsed();
        long max = memoryBean.getHeapMemoryUsage().getMax();
        double usage = (double) used / max;

        # 2. 判断健康状态
        if (usage > 0.9) {
            return Health.down()
                .withDetail("memory", String.format("%.2f%%", usage * 100))
                .withDetail("used", used)
                .withDetail("max", max)
                .build();
        }

        return Health.up()
            .withDetail("memory", String.format("%.2f%%", usage * 100))
            .build();
    }
}
```

## ⚠️ 日志最佳实践

### 1. 日志级别规范

```java
# TRACE：最详细的日志，追踪问题
log.trace("进入方法，参数：{}", params);

# DEBUG：调试信息
log.debug("查询数据库，SQL：{}", sql);

# INFO：关键业务流程
log.info("订单创建成功，orderNo：{}", orderNo);
log.info("支付成功，orderNo：{}, amount：{}", orderNo, amount);

# WARN：潜在问题
log.warn("库存不足，bookId：{}, stock：{}", bookId, stock);
log.warn("Redis连接超时，使用缓存");

# ERROR：错误，需要处理
log.error("订单创建失败，userId：{}", userId, e);
```

### 2. 敏感信息脱敏

```java
// 脱敏工具类
public class LogMaskUtil {
    private static final String MASK = "****";

    public static String maskPhone(String phone) {
        if (phone == null || phone.length() != 11) {
            return phone;
        }
        return phone.substring(0, 3) + MASK + phone.substring(7);
    }

    public static String maskIdCard(String idCard) {
        if (idCard == null || idCard.length() != 18) {
            return idCard;
        }
        return idCard.substring(0, 6) + MASK + idCard.substring(14);
    }

    public static String maskPassword(String password) {
        return MASK;  # 密码完全隐藏
    }
}

// 使用
log.info("用户登录成功，phone：{}", LogMaskUtil.maskPhone(phone));
```

### 3. 异常日志完整

```java
try {
    # 业务逻辑
} catch (Exception e) {
    # 完整的异常日志
    log.error("订单创建失败，userId：{}, orderNo：{}, error：{}",
        userId, orderNo, e.getMessage(), e);

    # 包含：
    # - 上下文信息（userId, orderNo）
    # - 异常消息（e.getMessage()）
    # - 异常堆栈（e）
}
```

## 📊 Django vs Spring Boot 对比

| 功能 | Django | Spring Boot | 难度 |
|------|--------|-------------|------|
| 日志框架 | logging模块 | SLF4J + Logback | 相当 |

| 请求追踪 | 自定义中间件 | MDC + Filter | 相当 |
| 应用监控 | django-prometheus | Actuator | Spring Boot更完善 |
| 可视化 | 同样用Grafana | 同样用Grafana | 相当 |

**Django优势**：
- logging模块配置简单

**Spring Boot优势**：
- Actuator功能强大
- MDC原生支持
- Prometheus集成好
