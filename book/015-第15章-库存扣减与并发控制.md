## 🤔 为什么需要并发控制？

### 真实场景：秒杀系统的崩溃

```
某电商平台《三体》秒杀：
┌─────────────────────────────────────────────────────────────┐
│  配置：库存10本，1000人抢                                   │
│                                                             │
│  时间线：                                                   │
│  10:00:00.000  用户1-1000同时点击"立即购买"                 │
│  10:00:00.100  检查库存：10本（1000个请求都看到库存10本）  │
│  10:00:00.200  开始扣库存                                  │
│  10:00:01.000  数据库CPU：100%                             │
│  10:00:02.000  数据库锁等待...                              │
│  10:00:05.000  最终结果：售出50本（超卖40本！）             │
│                                                             │
│  原因：                                                     │
│  1. 多个事务同时读取库存=10                                │
│  2. 都认为可以购买                                         │
│  3. 最终扣减成负数                                          │
│                                                             │
│  后果：                                                     │
│  - 商家亏钱（要发货40本）                                   │
│  - 用户投诉（买到后被告知没货）                             │
│  - 平台信誉受损                                             │
└─────────────────────────────────────────────────────────────┘
```

**🤔 为什么会超卖？**

```
数据库层面的原因：
UPDATE book SET stock = stock - 1 WHERE id = 123

问题：这不是原子操作！
分解为：
1. 读取 stock = 10
2. 计算 stock = 10 - 1 = 9
3. 写入 stock = 9

如果两个事务同时执行：
事务A：读取10 → 计算9 → (准备写入)
事务B：读取10 → 计算9 → (准备写入)
事务A：写入9
事务B：写入9  # 覆盖了事务A的结果！

最终：stock = 9（应该是8）
```

## 第二步：悲观锁实现

### 问题：什么时候用悲观锁？

```
适用场景：
1. 并发量低（<100 QPS）
2. 写操作频繁
3. 冲突概率高
4. 需要强一致性

淘书网适用场景：
- 后台管理员修改库存
- 批量导入商品
- 库存盘点
```

### 代码实现：悲观锁

```java
// service/InventoryService.java
package com.taoshu.service;

import com.taoshu.entity.Book;
import com.taoshu.repository.BookRepository;
import jakarta.persistence.LockModeType;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.data.jpa.repository.Lock;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class InventoryService {

    @Autowired
    private BookRepository bookRepository;

    # 悲观锁扣减库存
    @Transactional(rollbackFor = Exception.class)
    public void decreaseStockWithPessimisticLock(Long bookId, Integer quantity) {
        # 使用悲观锁查询（SELECT ... FOR UPDATE）
        Book book = bookRepository.findWithPessimisticLock(bookId)
            .orElseThrow(() -> new RuntimeException("商品不存在"));

        # 检查库存
        if (book.getStockQuantity() < quantity) {
            throw new RuntimeException("库存不足");
        }

        # 扣减库存
        book.setStockQuantity(book.getStockQuantity() - quantity);
        book.setSoldQuantity(book.getSoldQuantity() + quantity);
        bookRepository.save(book);
    }
}
```

```java
// repository/BookRepository.java
@Repository
public interface BookRepository extends JpaRepository<Book, Long> {

    /**
     * 使用悲观锁查询（SELECT FOR UPDATE）
     */
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT b FROM Book b WHERE b.id = :bookId")
    Optional<Book> findWithPessimisticLock(@Param("bookId") Long bookId);
}
```

**SQL执行过程**

```sql
-- 开启事务
BEGIN;

-- 使用悲观锁查询（其他事务等待）
SELECT * FROM book WHERE id = 123 FOR UPDATE;
-- 此时，其他事务想查询这条记录，必须等待

-- 检查库存（应用层）
-- stock_quantity = 10, quantity = 1 → 通过

-- 更新库存
UPDATE book SET stock_quantity = 9 WHERE id = 123;

-- 提交事务（释放锁）
COMMIT;
-- 其他事务才能继续
```

**性能分析**

```
并发场景：100个用户同时购买

悲观锁：
用户1：获取锁 → 扣库存 → 释放锁（耗时50ms）
用户2：等待50ms → 获取锁 → 扣库存 → 释放锁（耗时50ms）
用户3：等待100ms → ...
...
总耗时：100 × 50ms = 5000ms

问题：串行执行，性能差
```

## 第三步：乐观锁优化

### 问题：如何提高并发性能？

```
乐观锁原理：
不提前加锁，让多个事务同时执行
提交时检查版本号，如果版本号变了，说明数据被修改过
放弃本次修改，重试

优点：
1. 不需要等待锁
2. 并发性能好
3. 适合冲突概率低的场景

缺点：
1. 冲突时需要重试
2. 高并发场景下重试次数多
```

### 代码实现：乐观锁重试

```java
// service/InventoryService.java
@Service
public class InventoryService {

    private static final int MAX_RETRY = 3;  # 最大重试次数
    private static final long RETRY_DELAY = 10;  # 重试间隔（毫秒）

    # 乐观锁扣减库存（带重试）
    @Transactional(rollbackFor = Exception.class)
    public void decreaseStockWithOptimisticLock(Long bookId, Integer quantity) {
        int retry = 0;

        while (retry < MAX_RETRY) {
            try {
                # 1. 查询商品（带版本号）
                Book book = bookRepository.findById(bookId)
                    .orElseThrow(() -> new RuntimeException("商品不存在"));

                # 2. 检查库存
                if (book.getStockQuantity() < quantity) {
                    throw new RuntimeException("库存不足");
                }

                # 3. 使用乐观锁更新
                int updated = bookRepository.decreaseStock(
                    bookId,
                    quantity,
                    book.getVersion()
                );

                if (updated > 0) {
                    # 成功
                    return;
                }

                # 更新失败，版本号冲突
                retry++;
                if (retry < MAX_RETRY) {
                    Thread.sleep(RETRY_DELAY);
                }

            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                throw new RuntimeException("扣减库存失败");
            }
        }

        throw new RuntimeException("商品太火爆，请稍后重试");
    }
}
```

**执行过程分析**

```
时间线（3个用户同时购买）：

T0（0ms）：
用户A：查询 stock=10, version=5
用户B：查询 stock=10, version=5
用户C：查询 stock=10, version=5

T1（10ms）：
用户A：UPDATE ... WHERE version=5 → 成功，version=6
用户B：UPDATE ... WHERE version=5 → 失败（version已是6）
用户C：UPDATE ... WHERE version=5 → 失败（version已是6）

T2（20ms）：
用户B：重试，查询 stock=9, version=6
用户C：重试，查询 stock=9, version=6

T3（30ms）：
用户B：UPDATE ... WHERE version=6 → 成功，version=7
用户C：UPDATE ... WHERE version=6 → 失败（version已是7）

T4（40ms）：
用户C：重试，查询 stock=8, version=7
用户C：UPDATE ... WHERE version=7 → 成功，version=8

总耗时：40ms（悲观锁需要150ms）
```

## 第四步：Redis预减库存

### 问题：高并发场景下，数据库仍然是瓶颈

```
问题分析：
即使使用乐观锁，高并发下：
1. 大量请求打到数据库
2. 数据库连接池耗尽
3. 响应变慢
4. 用户体验差

解决思路：
将库存预热到Redis，所有请求先扣Redis
异步同步到数据库
```

### 架构设计

```

│      ↓                                                      │
│  限流（10000 QPS）                                           │
│      ↓                                                      │

│      ↓                                                      │
│  ├─ 有库存？                                                │
│  │   ├─ 是 → Redis扣减（原子操作）                          │
│  │   │         ↓                                            │
│  │   │   发送消息到MQ（异步）                               │
│  │   │         ↓                                            │

│  1. Redis操作快（0.1ms vs 数据库50ms）                      │
│  2. 原子操作保证不超卖                                      │
```

### 代码实现：Redis预减库存

#### 1. 库存预热

```java
// service/InventoryWarmupService.java
package com.taoshu.service;

import com.taoshu.entity.Book;
import com.taoshu.repository.BookRepository;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Service;
import java.util.List;

@Service
public class InventoryWarmupService {

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Autowired
    private BookRepository bookRepository;

    private static final String STOCK_KEY_PREFIX = "stock:book:";

    # 库存预热（应用启动时执行）
    @PostConstruct
    public void warmupStock() {
        List<Book> books = bookRepository.findAll();
        for (Book book : books) {
            String key = STOCK_KEY_PREFIX + book.getId();
            # 初始化库存到Redis
            redisTemplate.opsForValue().set(
                key,
                book.getStockQuantity().toString()
            );
        }
    }

    # 定时同步库存（每分钟）
    @Scheduled(fixedRate = 60000)
    public void syncStock() {
        # 从数据库读取最新库存
        List<Book> books = bookRepository.findAll();
        for (Book book : books) {
            String key = STOCK_KEY_PREFIX + book.getId();
            # 只更新库存较少的（防止缓存穿透）
            if (book.getStockQuantity() < 100) {
                redisTemplate.opsForValue().set(
                    key,
                    book.getStockQuantity().toString()
                );
            }
        }
    }
}
```

#### 2. Redis原子扣减

```java
// service/RedisInventoryService.java
@Service
public class RedisInventoryService {

    @Autowired
    private StringRedisTemplate redisTemplate;

    @Autowired
    private RabbitTemplate rabbitTemplate;

    private static final String STOCK_KEY_PREFIX = "stock:book:";
    private static final String ORDER_QUEUE = "order.queue";

    /**
     * Redis预减库存（原子操作）
     * @param bookId 商品ID
     * @param quantity 数量
     * @return true=成功, false=库存不足
     */
    public boolean decreaseStock(Long bookId, Integer quantity) {
        String key = STOCK_KEY_PREFIX + bookId;

        # 使用Lua脚本保证原子性
        String script =
            "local stock = tonumber(redis.call('GET', KEYS[1])) " +
            "if stock >= tonumber(ARGV[1]) then " +
            "    redis.call('DECRBY', KEYS[1], ARGV[1]) " +
            "    return 1 " +
            "else " +
            "    return 0 " +
            "end";

        DefaultRedisScript<Long> redisScript = new DefaultRedisScript<>(script, Long.class);

        Long result = redisTemplate.execute(
            redisScript,
            Collections.singletonList(key),
            quantity.toString()
        );

        if (result == 1) {
            # 扣减成功，发送消息到MQ
            OrderMessage message = new OrderMessage();
            message.setBookId(bookId);
            message.setQuantity(quantity);
            rabbitTemplate.convertAndSend(ORDER_QUEUE, message);
            return true;
        }

        return false;
    }
}
```

**🤔 为什么用Lua脚本？**

```
不用Lua脚本（有问题）：
1. GET stock:book:123 → 返回10
2. 应用层判断：10 >= 1，可以扣减
3. （此时另一个请求也执行了步骤1-2）
4. DECRBY stock:book:123 1 → 库存=9
5. （另一个请求也执行了步骤4）
6. 最终库存=8（应该是9，被多扣了1）

使用Lua脚本（原子操作）：
EXECUTE script
- GET stock:book:123 → 10
- 判断 10 >= 1 → 通过
- DECRBY stock:book:123 1 → 库存=9
- 返回 1（成功）

整个过程在Redis服务器端完成，不会被其他请求打断
```

### 代码实现：秒杀接口

```java
// controller/SeckillController.java
@RestController
@RequestMapping("/api/seckill")
public class SeckillController {

    @Autowired
    private RedisInventoryService redisInventoryService;

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String SECKILL_LOCK_KEY = "seckill:lock:";
    private static final String USER_PURCHASE_KEY = "seckill:user:";

    # 秒杀下单
    @PostMapping("/{bookId}")
    public Result<?> seckill(
            @PathVariable Long bookId,
            @RequestHeader("userId") Long userId) {

        # 1. 检查用户是否已购买
        String userKey = USER_PURCHASE_KEY + bookId + ":" + userId;
        Boolean hasPurchased = redisTemplate.hasKey(userKey);
        if (Boolean.TRUE.equals(hasPurchased)) {
            return Result.error("每人限购1件");
        }

        # 2. 分布式锁（防止重复下单）
        String lockKey = SECKILL_LOCK_KEY + userId;
        Boolean locked = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS);

        if (!Boolean.TRUE.equals(locked)) {
            return Result.error("请勿重复提交");
        }

        try {
            # 3. Redis扣减库存
            boolean success = redisInventoryService.decreaseStock(bookId, 1);
            if (!success) {
                return Result.error("库存不足");
            }

            # 4. 标记用户已购买
            redisTemplate.opsForValue().set(userKey, "1", 1, TimeUnit.HOURS);

            # 5. 返回"处理中"（轮询查询结果）
            return Result.success("正在处理，请稍后查询订单");

        } finally {
            # 释放锁
            redisTemplate.delete(lockKey);
        }
    }

    # 查询秒杀结果
    @GetMapping("/result/{taskId}")
    public Result<?> getSeckillResult(@PathVariable String taskId) {
        # 查询订单状态
        Order order = orderRepository.findByTaskId(taskId);
        if (order == null) {
            return Result.success("处理中...");
        }
        return Result.success(OrderResponse.fromEntity(order));
    }
}
```

```java
public class RateLimiter {

    private final Map<Long, RateLimiter> limiters = new ConcurrentHashMap<>();

    public boolean tryAcquire(Long userId) {
        RateLimiter limiter = limiters.computeIfAbsent(
            userId,
```

| 悲观锁 | `select_for_update()` | `@Lock(PESSIMISTIC_WRITE)` | 相当 |
| 乐观锁 | `F对象 + version` | `@Version` | 相当 |
| Redis预减 | 手动实现 | RedisTemplate | 相当 |
| 限流 | Django Ratelimit | Resilience4j | 相当 |
| 分布式锁 | Redis分布式锁 | Redisson | Spring Boot更完善 |

**Django优势**：
- `select_for_update()` 语法简洁

**Spring Boot优势**：
- Redisson功能完善
- Resilience4j集成好
- MQ集成简单

## 🎯 本章小结

| 知识点 | 为什么需要 | 怎么做 |
|--------|-----------|--------|
| **并发控制** | 防止超卖、保证一致性 | 悲观锁 / 乐观锁 / Redis |
| **悲观锁** | 绝对安全、适合低并发 | SELECT FOR UPDATE |
| **乐观锁** | 高性能、适合中并发 | @Version + 重试 |
| **Redis预减** | 保护数据库、高性能 | Lua脚本 + MQ |
| **秒杀系统** | 应对超高并发 | 限流 + Redis + MQ |
| **分布式锁** | 防止重复操作 | Redis SET NX |

**记忆口诀**：
- **低并发悲观锁**，**中并发乐观锁**
- **高并发用Redis**，**秒杀限流加队列**
- **Lua保原子**，**MQ削峰填谷**
- **分段减竞争**，**本地缓存快**
