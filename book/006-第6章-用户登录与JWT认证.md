# 第6章：用户登录与JWT认证

## 🤔 为什么需要JWT认证？

> **小王的疑惑**：Django用Session就能记住用户，为什么Spring Boot要用这么复杂的JWT？

### 从一个真实的扩容问题说起

**场景：某公司用Session认证，业务快速增长后遇到的问题**

```
初期（1台服务器）：
├── Django Session存储在数据库
├── 用户登录后，Session存入数据库
└── 下次请求用Session ID查询数据库
→ 没问题

发展期（3台服务器 + 负载均衡）：
├── 用户A登录到服务器1
├── Session存在服务器1的数据库
├── 第二次请求被负载均衡到服务器2
└── 服务器2找不到Session！→ 用户被登出！❌

解决方式：Session共享（Redis）
├── 所有服务器的Session存到Redis
├── 负载均衡没问题了
└── 但是：每次请求都要查询Redis → 性能下降
```

**JWT的解决方案**：

```
JWT认证流程：
1. 用户登录 → 服务器生成Token
2. Token包含用户信息（加密）
3. Token返回给客户端（客户端存储）
4. 每次请求带上Token
5. 服务器验证Token → 无需查询数据库

优势：
✅ 无状态：服务器不存储Session
✅ 可扩展：多台服务器无需共享Session
✅ 性能好：不需要每次查询数据库或Redis
✅ 跨域友好：适合前后端分离
```

### 🎯 Session vs JWT 对比

| 特性 | Session | JWT |
|------|--------|-----|
| **存储位置** | 服务器（数据库/Redis） | 客户端（localStorage） |
| **状态** | 有状态 | 无状态 |
| **扩展性** | 需要Session共享 | 天然支持分布式 |
| **性能** | 每次查询存储 | 只验证签名 |
| **安全性** | 服务器控制 | 可能被窃取 |
| **大小限制** | 无 | Token不宜过大 |
| **注销** | 销毁Session | Token无法主动失效 |

**生活类比**：

```
Session = 剧院寄存柜
→ 存包（登录）
→ 取票（验证）
→ 取包（登出）
→ 必须回同一个剧院（同一台服务器）

JWT = 电子票（二维码）
→ 购票（登录）
→ 扫码（验证）
→ 不需要寄存柜
→ 可以在任何影院使用
```

### 6.1 JWT是什么？

#### 🎯 JWT的结构

**JWT = JSON Web Token**

```
JWT格式：xxxxx.yyyyy.zzzzz

xxxxx = Header（头部）
yyyyy = Payload（负载）
zzzzz = Signature（签名）
```

**实际例子**：

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOjEsInBob25lIjoiMTM4MDAxMzgwMDAifQ.QT6f...

解码后：
Header:
{
  "alg": "HS256",  # 算法
  "typ": "JWT"     # 类型
}

Payload:
{
  "userId": 1,
  "phone": "13800138000"
  # 可以包含用户信息
}

Signature:
# 用密钥对Header.Payload进行签名
# 验证时用同样的密钥验证签名
```

**为什么JWT可以不存储用户信息？**

```
JWT是自包含的：
→ Payload包含用户信息
→ Signature保证信息不被篡改
→ 服务器只需要验证签名
→ 不需要查询数据库
```

### 6.2 JWT配置与工具类

#### 🤔 为什么不能直接用String拼接JWT？

**错误的JWT生成**：

```java
// ❌ 错误：手动拼接JWT
public String generateToken(Long userId, String phone) {
    String header = "{\"alg\":\"HS256\",\"typ\":\"JWT\"}";
    String payload = "{\"userId\":" + userId + ",\"phone\":\"" + phone + "\"}";
    String signature = HMACSHA256(header + "." + payload, secret);

    return header + "." + payload + "." + signature;
}
// 问题：
// 1. 容易出错
// 2. 没有处理Base64编码
// 3. 签名算法容易实现错
```

**正确的做法：用成熟的库**：

```xml
<!-- pom.xml -->
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-api</artifactId>
    <version>0.12.3</version>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-impl</artifactId>
    <version>0.12.3</version>
    <scope>runtime</scope>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-jackson</artifactId>
    <version>0.12.3</version>
    <scope>runtime</scope>
</dependency>
```

```java
// JwtUtil.java
@Component
@RequiredArgsConstructor
public class JwtUtil {

    private final JwtProperties jwtProperties;

    public String generateToken(Long userId, String phone) {
        // 用成熟的库生成JWT
        return Jwts.builder()
            .claim("userId", userId)
            .claim("phone", phone)
            .subject(String.valueOf(userId))
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + jwtProperties.getExpiration()))
            .signWith(getSecretKey())
            .compact();
    }

    public Long getUserIdFromToken(String token) {
        Claims claims = Jwts.parser()
            .verifyWith(getSecretKey())
            .build()
            .parseSignedClaims(token)
            .getPayload();
        return Long.valueOf(claims.get("userId", String.class));
    }

    public boolean validateToken(String token) {
        try {
            Jwts.parser()
                .verifyWith(getSecretKey())
                .build()
                .parseSignedClaims(token);
            return true;
        } catch (Exception e) {
            return false;
        }
    }

    private SecretKey getSecretKey() {
        return Keys.hmacShaKeyFor(
            jwtProperties.getSecret().getBytes(StandardCharsets.UTF_8)
        );
    }
}
```

### 6.3 登录接口实现

#### 🎯 登录流程详解

```
用户提交登录
    ↓
1️⃣ 验证用户名密码
    ↓
2️⃣ 检查用户状态（是否被封禁）
    ↓
3️⃣ 生成JWT Token
    ↓
4️⃣ 返回Token和用户信息
```

**登录Service**：

```java
@Service
@RequiredArgsConstructor
public class AuthService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;
    private final JwtUtil jwtUtil;

    public String login(UserLoginRequest request) {
        // 1. 查询用户
        User user = userRepository.findByPhone(request.getPhone())
            .orElseThrow(() -> new BusinessException("用户不存在"));

        // 2. 验证密码
        if (!passwordEncoder.matches(request.getPassword(), user.getPassword())) {
            throw new BusinessException("密码错误");
        }

        // 3. 检查状态
        if (user.getStatus() == 0) {
            throw new BusinessException("账号已被禁用");
        }

        // 4. 更新最后登录时间
        user.setLastLoginAt(LocalDateTime.now());
        userRepository.save(user);

        // 5. 生成Token
        return jwtUtil.generateToken(user.getId(), user.getPhone());
    }
}
```

**登录Controller**：

```java
@RestController
@RequestMapping("/api/user")
@RequiredArgsConstructor
public class AuthController {

    private final AuthService authService;

    @PostMapping("/login")
    public Result<LoginResponse> login(@Valid @RequestBody UserLoginRequest request) {
        // 1. 登录验证，生成Token
        String token = authService.login(request);

        // 2. 从Token中解析用户ID（临时方案）
        // TODO: 直接在AuthService返回用户信息
        Long userId = getUserIdFromPhone(request.getPhone());
        User user = authService.getUserById(userId);

        // 3. 构造响应
        LoginResponse response = new LoginResponse();
        response.setToken(token);
        response.setTokenType("Bearer");
        response.setUser(UserResponse.fromEntity(user));

        return Result.success("登录成功", response);
    }
}
```

### 6.4 JWT认证拦截器

#### 🤔 为什么需要拦截器？

**场景：保护需要登录才能访问的接口**

```python
# Django：用装饰器
@login_required
def get_profile(request):
    return Response({'user': request.user})

# Django中间件自动检查Session
# 没有Session → 重定向到登录页
```

```java
// Java：用拦截器
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @PostMapping
    public Result<OrderResponse> createOrder(@RequestBody CreateOrderRequest request) {
        // ❌ 问题：如何知道用户是否登录？
        Long userId = getCurrentUserId();  // 从哪里获取？
        // ...
    }
}
```

**拦截器的解决方案**：

```java
@Component
@RequiredArgsConstructor
public class JwtAuthInterceptor implements HandlerInterceptor {

    private final JwtUtil jwtUtil;

    @Override
    public boolean preHandle(HttpServletRequest request,
                           HttpServletResponse response,
                           Object handler) throws Exception {
        // 1. 从请求头获取Token
        String authHeader = request.getHeader("Authorization");

        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            sendErrorResponse(response, 401, "未认证或Token格式错误");
            return false;
        }

        // 2. 提取Token
        String token = authHeader.substring(7);  // 去掉"Bearer "

        // 3. 验证Token
        if (!jwtUtil.validateToken(token)) {
            sendErrorResponse(response, 401, "Token无效或已过期");
            return false;
        }

        // 4. 提取用户信息并存入请求属性
        Long userId = jwtUtil.getUserIdFromToken(token);
        String phone = jwtUtil.getPhoneFromToken(token);
        request.setAttribute("userId", userId);
        request.setAttribute("phone", phone);

        return true;  // 放行，继续执行Controller
    }

    private void sendErrorResponse(HttpServletResponse response,
                                     int status, String message) throws IOException {
        response.setStatus(status);
        response.setContentType("application/json;charset=UTF-8");
        response.getWriter().write(
            "{\"code\":" + status + ",\"message\":\"" + message + "\"}"
        );
    }
}
```

**配置拦截器**：

```java
@Configuration
@RequiredArgsConstructor
public class WebMvcConfig implements WebMvcConfigurer {

    private final JwtAuthInterceptor jwtAuthInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(jwtAuthInterceptor)
                .addPathPatterns("/api/**")  // 拦截所有API请求
                .excludePathPatterns(
                        "/api/user/register",  // 排除注册
                        "/api/user/login",     // 排除登录
                        "/api/user/send-code", // 排除发送验证码
                        "/api/hello/**"        // 排除测试接口
                );
    }
}
```

### 6.5 在Controller中获取当前用户

#### 🎯 便捷工具类

```java
// UserContext.java
public class UserContext {

    /**
     * 获取当前登录用户ID
     * 对比Django: request.user.id
     */
    public static Long getUserId(HttpServletRequest request) {
        Object userId = request.getAttribute("userId");
        if (userId == null) {
            throw new BusinessException("未登录");
        }
        return Long.valueOf(userId.toString());
    }

    /**
     * 获取当前登录用户手机号
     * 对比Django: request.user.username
     */
    public static String getPhone(HttpServletRequest request) {
        Object phone = request.getAttribute("phone");
        if (phone == null) {
            throw new BusinessException("未登录");
        }
        return phone.toString();
    }
}
```

**使用**：

```java
@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;

    @PostMapping
    public Result<OrderResponse> createOrder(@RequestBody CreateOrderRequest request,
                                              HttpServletRequest httpRequest) {
        // 获取当前用户ID
        Long userId = UserContext.getUserId(httpRequest);

        // 创建订单
        OrderResponse order = orderService.createOrder(request, userId);

        return Result.success(order);
    }
}
```

### 6.6 Token刷新机制

#### 🤔 为什么Token需要刷新？

**问题：Token的有效期**

```
Token有效期太长（如30天）：
→ 容易被窃取后滥用
→ 用户修改密码后Token仍然有效
→ 安全风险

Token有效期太短（如1小时）：
→ 用户频繁需要重新登录
→ 体验不好
```

**解决方案：RefreshToken + Access Token**

```
Access Token（访问令牌）：
→ 有效期短（15-30分钟）
→ 包含用户基本信息
→ 用于API请求
→ 过期后需要重新登录

RefreshToken（刷新令牌）：
→ 有效期长（7-30天）
→ 只用于刷新Access Token
→ 存储在HttpOnly Cookie中（更安全）
```

**简化实现（只用Access Token）**：

```java
// 当前实现：使用较长的Access Token有效期
@Value("${jwt.expiration}")
private Long expiration;  // 24小时

// 如果用户觉得不安全，可以：
// 1. 缩短Access Token有效期到30分钟
// 2. 添加RefreshToken机制
// 3. 支持Token主动注销（黑名单）
```

### 6.7 注销登录

#### 🤔 JWT是无状态的，怎么实现注销？

**问题**：

```
Session认证：
→ 注销 = 销毁Session
→ 服务器端主动失效
→ 简单

JWT认证：
→ Token在客户端
→ 服务器不存储
→ 注销？Token仍然有效！
```

**解决方案**：

```java
// 方案1：短期Token（当前方案）
// Token有效期24小时
// 用户点击注销只是清除客户端Token
// 服务器端Token在24小时后自动失效
// → 简单但有安全风险

// 方案2：Token黑名单（生产环境推荐）
@Service
public class TokenBlacklistService {

    private final RedisTemplate<String, String> redisTemplate;

    public void addToBlacklist(String token) {
        // 获取Token剩余有效时间
        Long expiration = jwtUtil.getExpiration(token);

        // 存入Redis，到期自动删除
        redisTemplate.opsForValue()
            .set("token:blacklist:" + token, "true",
                    expiration, TimeUnit.SECONDS);
    }

    public boolean isBlacklisted(String token) {
        return Boolean.TRUE.equals(
            redisTemplate.opsForValue().get("token:blacklist:" + token)
        );
    }
}

// 登出时
@PostMapping("/logout")
public Result<Void> logout(HttpServletRequest request) {
    String token = extractToken(request);
    tokenBlacklistService.addToBlacklist(token);

    // 清除客户端Token
    return Result.success("退出成功");
}

// 拦截器验证时检查黑名单
if (tokenBlacklistService.isBlacklisted(token)) {
    sendErrorResponse(response, 401, "Token已失效");
    return false;
}
```

### 6.8 前端Token使用

#### 📱 前端如何使用Token

```javascript
// React示例
import axios from 'axios';

// 登录
const login = async (phone, password) => {
  const response = await axios.post('/api/user/login', {
    phone,
    password
  });

  // 存储Token
  const { token } = response.data.data;
  localStorage.setItem('token', token);

  // 配置axios默认携带Token
  axios.defaults.headers.common['Authorization'] = `Bearer ${token}`;
};

// 自动拦截器
axios.interceptors.request.use(config => {
  const token = localStorage.getItem('token');
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// 请求示例
const getOrders = async () => {
  const response = await axios.get('/api/orders');
  // Token会自动添加到请求头
  return response.data;
};

// 登出
const logout = () => {
  localStorage.removeItem('token');
  delete axios.defaults.headers.common['Authorization'];
  // 清除Token，后续请求会返回401
};
```

### 6.9 本章小结

#### 🎯 核心概念对比

| 概念 | Session | JWT |
|------|--------|-----|
| **存储位置** | 服务器 | 客户端 |
| **状态** | 有状态 | 无状态 |
| **分布式支持** | 需要Session共享 | 天然支持 |
| **性能** | 每次查询存储 | 只验证签名 |
| **安全性** | 服务器控制 | 可能被窃取 |
| **注销** | 销毁Session | 需要黑名单机制 |

#### 💡 JWT的优势

1. **无状态**：服务器不存储用户信息
2. **可扩展**：天然支持分布式部署
3. **跨域**：适合前后端分离
4. **移动端友好**：移动应用也可以使用

#### ⚠️ JWT的注意事项

1. **不要在Token中存储敏感信息**（如密码）
2. **Token要设置合理的过期时间**
3. **使用HTTPS传输**（防止被窃取）
4. **考虑实现RefreshToken机制**
