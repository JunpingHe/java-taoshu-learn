# 第7章：用户信息管理

## 🤔 为什么需要用户信息管理功能？

> **小王的疑惑**：注册登录都有了，用户信息管理不就是查查改改吗？有什么好讲的？

### 从一个用户体验问题说起

**场景：用户想修改个人信息时遇到的困扰**

```
问题1：信息展示不完整
→ 用户头像、注册时间看不到
→ 用户不知道自己是什么时候注册的

问题2：信息修改不友好
→ 想改昵称 → 需要重新登录？
→ 想改手机号 → 需要验证旧手机号？
→ 想改密码 → 不知道怎么改

问题3：头像上传太复杂
→ 不知道支持什么格式
→ 不知道文件大小限制
→ 上传失败也不知道为什么

问题4：隐私问题
→ 其他用户能看到我的手机号吗？
→ 我的购买记录会暴露吗？
```

### 🎯 完善的用户信息管理应该包含什么？

```
✅ 基础信息查看

✅ 隐私设置
  → 是否公开手机号
  → 是否公开购买记录

✅ 账号安全
  → 修改密码
  → 查看登录记录
```

### 7.1 获取用户信息

#### 🤔 为什么需要限制用户只能查看自己的信息？

**场景：没有权限控制的问题**

```python
# Django反面教材
@api_view(['GET'])
def get_user(request, user_id):
    # ❌ 问题：任何人都可以查询任何人的信息
    user = User.objects.get(id=user_id)
    serializer = UserSerializer(user)
    return Response(serializer.data)
    # 攻击者可以遍历所有用户ID获取信息
```

```java
// Java反面教材
@GetMapping("/{id}")
public Result<UserResponse> getUser(@PathVariable Long id) {
    // ❌ 问题：任何人都可以查询任何人的信息
    User user = userRepository.findById(id)
        .orElseThrow(() -> new BusinessException("用户不存在"));
    return Result.success(UserResponse.fromEntity(user));
}
```

**攻击示例**：

```bash
# 攻击者遍历用户ID
for i in range(1, 10000):
    curl http://example.com/api/user/$i

# 结果：获取大量用户信息（手机号、昵称等）
# 数据泄露！
```

---

#### ✅ 正确做法：权限验证

```java
@GetMapping("/{id}")
public Result<UserResponse> getUser(@PathVariable Long id,
                                  HttpServletRequest request) {
    // 1. 获取当前登录用户ID
    Long currentUserId = UserContext.getUserId(request);

    // 2. 查询目标用户
    User user = userRepository.findById(id)
        .orElseThrow(() -> new BusinessException("用户不存在"));

    // 3. 权限检查：只能查看自己的信息
    if (!currentUserId.equals(id)) {
        throw new BusinessException("无权查看其他用户信息");
    }

    return Result.success(UserResponse.fromEntity(user));
}
```

**更好的做法：创建权限验证注解**

```java
// 自定义注解：只能查看自己的信息
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @MustBeOwnerOrAdmin {
}

// 使用Aspect切面验证
@Aspect
@Component
public class PermissionAspect {

    @Around("@annotation(MustBeOwnerOrAdmin)")
    public Object checkPermission(ProceedingJoinPoint joinPoint) throws Throwable {
        // 获取方法参数
        Long userId = (Long) Arrays.stream(joinPoint.getArgs())
            .filter(arg -> arg instanceof Long)
            .findFirst()
            .orElse(null);

        // 获取当前用户
        HttpServletRequest request = getCurrentRequest();
        Long currentUserId = UserContext.getUserId(request);

        // 检查权限
        if (!currentUserId.equals(userId) && !isAdmin(currentUserId)) {
            throw new BusinessException("无权限");
        }

        return joinPoint.proceed();
    }
}

// 使用
@GetMapping("/{id}")
@MustBeOwnerOrAdmin
public Result<UserResponse> getUser(@PathVariable Long id) {
    // 权限检查由注解自动完成
    User user = userService.getById(id);
    return Result.success(UserResponse.fromEntity(user));
}
```

### 7.2 修改用户信息

#### 🤔 为什么修改信息需要验证？

**场景：恶意修改他人信息**

```
问题1：CSRF攻击
→ 用户访问恶意网站时
→ 网站用用户的身份发起修改请求
→ 用户信息被恶意修改

问题2：中间人攻击
→ 攻击者拦截修改请求
→ 修改参数（如手机号）
→ 信息被窃取

问题3：重放攻击
→ 攻击者截获修改请求
→ 多次重放（如修改手机号）
→ 额度消费他人手机号
```

**防护措施**：

```
✅ JWT认证：确保请求来自合法用户
✅ 权限检查：只能修改自己的信息
✅ 敏感操作二次验证：修改手机号需要验证码
✅ 参数验证：确保数据格式正确
✅ 限流：防止暴力攻击
```

---

#### 📝 修改昵称接口

```java
@PatchMapping("/nickname")
public Result<UserResponse> updateNickname(
        @RequestBody UpdateNicknameRequest request,
        HttpServletRequest httpRequest) {

    // 1. 获取当前用户
    Long userId = UserContext.getUserId(httpRequest);

    // 2. 验证昵称格式
    if (request.getNickname() == null || request.getNickname().isBlank()) {
        return Result.badRequest("昵称不能为空");
    }

    if (request.getNickname().length() > 50) {
        return Result.badRequest("昵称最长50位");
    }

    // 3. 检查昵称是否包含敏感词
    if (containsSensitiveWord(request.getNickname())) {
        return Result.error("昵称包含敏感词");
    }

    // 4. 更新昵称
    User user = userService.updateNickname(userId, request.getNickname());

    return Result.success("昵称修改成功", UserResponse.fromEntity(user));
}

// Service
@Transactional
public User updateNickname(Long userId, String nickname) {
    User user = getById(userId);
    user.setNickname(nickname);
    return userRepository.save(user);
}

// DTO
@Data
public class UpdateNicknameRequest {
    @Size(max = 50, message = "昵称最长50位")
    private String nickname;
}
```

### 7.3 修改密码

#### 🤔 为什么修改密码需要输入旧密码？

**场景：用户离开电脑忘记锁定**

```
问题：用户离开电脑，别人趁机修改密码
→ 如果不需要旧密码
→ 攻击者直接修改密码
→ 原用户无法登录账号

解决：需要输入旧密码
→ 必须知道旧密码才能修改
→ 提供一层安全保护
```

**修改密码接口**：

```java
@PostMapping("/password")
public Result<Void> updatePassword(
        @RequestBody UpdatePasswordRequest request,
        HttpServletRequest httpRequest) {

    Long userId = UserContext.getUserId(httpRequest);

    // 1. 验证旧密码
    userService.verifyOldPassword(userId, request.getOldPassword());

    // 2. 验证新密码强度
    if (request.getNewPassword().length() < 6) {
        return Result.badRequest("新密码长度至少6位");
    }

    if (!request.getNewPassword().equals(request.getConfirmPassword())) {
        return Result.badRequest("两次密码不一致");
    }

    // 3. 更新密码
    userService.updatePassword(userId, request.getNewPassword());

    return Result.success("密码修改成功");
}

// Service
@Transactional
public void updatePassword(Long userId, String newPassword) {
    User user = getById(userId);

    // 加密新密码
    String encodedPassword = passwordEncoder.encode(newPassword);
    user.setPassword(encodedPassword);

    userRepository.save(user);
}
```

### 7.4 修改手机号

#### 🤔 为什么修改手机号需要双重验证？

**场景：手机号被窃取**

```
问题1：手机号被其他人注册
→ 真实用户无法注册

问题2：手机号过期被回收
→ 运营商把手机号分配给别人
→ 原用户无法接收验证码

问题3：修改手机号时
→ 新手机号已经被别人使用
→ 或原手机号无法接收验证码
```

**修改手机号流程**：

```
1️⃣ 验证当前手机号
   → 向当前手机号发送验证码
   → 输入验证码确认身份

2️⃣ 验证新手机号
   → 检查新手机号是否已被使用
   → 向新手机号发送验证码
   → 输入验证码确认可用

3️⃣ 更新手机号
   → 保存新手机号到数据库
   → 返回成功
```

**接口实现**：

```java
@PostMapping("/phone/change")
public Result<Void> changePhone(
        @RequestBody ChangePhoneRequest request,
        HttpServletRequest httpRequest) {

    Long userId = UserContext.getUserId(httpRequest);

    // 1. 验证当前手机号
    if (!smsService.verifyCode(request.getPhone(), request.getCurrentCode())) {
        return Result.error("当前手机号验证码错误");
    }

    // 2. 验证新手机号
    if (userRepository.existsByPhone(request.getNewPhone())) {
        return Result.error("新手机号已被使用");
    }

    if (!smsService.verifyCode(request.getNewPhone(), request.getNewCode())) {
        return Result.error("新手机号验证码错误");
    }

    // 3. 更新手机号
    userService.updatePhone(userId, request.getNewPhone());

    return Result.success("手机号修改成功");
}

// DTO
@Data
public class ChangePhoneRequest {

    @NotBlank
    private String phone;  // 当前手机号

    @NotBlank
    private String currentCode;  // 当前手机号验证码

    @NotBlank
    @Pattern(regexp = "^1\\d{10}$")
    private String newPhone;  // 新手机号

    @NotBlank
    private String newCode;  // 新手机号验证码
}
```

### 7.5 头像上传

#### 🤔 为什么需要限制上传文件的格式和大小？

**场景：不受限制的文件上传**

```
问题1：上传大文件
→ 用户上传10MB的图片
→ 服务器存储、带宽压力
→ 其他用户下载慢

问题2：上传错误格式
→ 用户上传.exe文件伪装成.jpg
→ 可能包含病毒
→ 消费者下载后中毒

问题3：图片过大
→ 10000x10000像素的图片
→ 显示慢、存储大
→ 加载体验差
```

**文件上传限制**：

```java
// application.yml
spring:
  servlet:
    multipart:
      enabled: true
      max-file-size: 5MB      # 单个文件最大5MB
      max-request-size: 10MB  # 一次请求最大10MB

app:
  upload:
    allowed-types: jpg,jpeg,png,gif,webp  # 允许的图片格式
    max-width: 2000      # 图片最大宽度
    max-height: 2000     # 图片最大高度
```

**上传接口**：

```java
@PostMapping("/avatar")
public Result<String> uploadAvatar(
        @RequestParam("file") MultipartFile file,
        HttpServletRequest httpRequest) {

    Long userId = UserContext.getUserId(httpRequest);

    // 1. 验证文件
    if (file.isEmpty()) {
        return Result.badRequest("请选择文件");
    }

    // 2. 验证文件类型
    String contentType = file.getContentType();
    if (!Arrays.asList("image/jpeg", "image/png", "image/gif", "image/webp").contains(contentType)) {
        return Result.badRequest("只支持jpg、png、gif、webp格式");
    }

    // 3. 验证文件大小
    if (file.getSize() > 5 * 1024 * 1024) {
        return Result.badRequest("文件大小不能超过5MB");
    }

    // 4. 生成文件名
    String originalFilename = file.getOriginalFilename();
    String extension = originalFilename.substring(originalFilename.lastIndexOf("."));
    String filename = UUID.randomUUID() + extension;

    // 5. 保存文件
    try {
        String path = uploadPath + filename;
        file.transferTo(new File(path));

        // 6. 更新用户头像
        String avatarUrl = "/uploads/" + filename;
        userService.updateAvatar(userId, avatarUrl);

        return Result.success("头像上传成功", avatarUrl);
    } catch (IOException e) {
        log.error("文件上传失败", e);
        return Result.error("文件上传失败");
    }
}
```

### 7.7 账号注销

#### 🤔 为什么需要"冷静期"注销？

**场景：用户冲动注销账号**

```
问题：用户一时冲动注销账号
→ 注销后无法恢复
→ 数据全部丢失
→ 用户后悔了

解决：冷却期机制
→ 注销申请 → 30天冷静期
→ 30天内可以取消注销
→ 30天后永久注销
```

**注销流程**：

```
1️⃣ 申请注销
   → 用户点击"申请注销"
   → 填写注销原因

2️⃣ 进入冷却期
   → 账号变为"待注销"状态
   → 30天内可以取消

3️⃒ 取消注销（冷却期内）
   → 用户点击"取消注销"
   → 账号恢复正常

4️⃒ 确认注销（冷却期后）
   → 系统发送确认邮件
   → 用户点击确认链接
   → 永久注销账号
```

**实现**：

```java
// Service
public void applyForCancellation(Long userId, String reason) {
    User user = getById(userId);

    // 检查条件
    if (hasUnpaidOrders(userId)) {
        throw new BusinessException("有待支付订单，无法注销");
    }

    if (hasActiveSellingItems(userId)) {
        throw new BusinessException("有在售商品，无法注销");
    }

    // 设置为"待注销"状态
    user.setStatus(UserStatus.PENDING_CANCELLATION);
    user.setCancellationReason(reason);
    user.setCancellationApplyTime(LocalDateTime.now());
    userRepository.save(user);

    // 发送确认邮件
    sendCancellationEmail(user);
}

// 定时任务：30天后执行注销
@Scheduled(cron = "0 0 2 * * ?")
public void processExpiredCancellations() {
    List<User> users = userRepository.findByStatusAndCancellationTimeBefore(
        UserStatus.PENDING_CANCELLATION,
        LocalDateTime.now().minusDays(30)
    );

    for (User user : users) {
        // 匿名化数据
        anonymizeUserData(user);

        // 标记为已注销
        user.setStatus(UserStatus.CANCELLED);
        userRepository.save(user);
    }
}
```

### 7.8 实用功能：查看登录记录

#### 🎯 为什么需要登录记录？

**场景：发现异常登录**

```
用户发现：
→ 昨天凌晨2点有人登录
→ 登录地点在国外
→ 购买记录有异常

查看登录记录：
→ 发现不是本人操作
→ 立即修改密码
→ 账号保住了
```

### 7.9 本章小结

#### 🎯 用户信息管理要点

| 功能 | 关键点 | 安全考虑 |
|------|--------|----------|
| **查询用户** | 只能查自己的信息 | 权限验证 |
| **修改昵称** | 敏感词过滤 | 长度限制 |
| **修改密码** | 需要旧密码 | 新密码强度检查 |
| **修改手机号** | 双重验证 | 防止劫持 |
| **上传头像** | 格式、大小限制 | 防止上传恶意文件 |
| **隐私设置** | 用户可控 | 默认最严格 |
| **注销账号** | 冷却期机制 | 防止误操作 |

#### 💡 最佳实践

1. **权限验证**：每个接口都要验证当前用户权限
2. **参数验证**：使用@Valid注解自动验证
3. **二次确认**：敏感操作（改密码、改手机号）需要验证
4. **日志记录**：记录关键操作，便于审计
5. **友好提示**：错误信息要清晰，告诉用户怎么做
