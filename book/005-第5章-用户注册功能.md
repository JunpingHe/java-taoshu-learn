# 第5章：用户注册功能

## 🤔 为什么需要这么复杂的注册流程？

> **小王的疑惑**：注册不就是"收数据 → 存数据库"吗？为什么需要DTO、Service、Repository这么多层？

### 从一个安全事故说起

**场景：2019年某公司用户数据泄露事件**

```python
# 有问题的Django代码（反面教材）
def register(request):
    # ❌ 问题1：没有验证手机号格式
    phone = request.data.get('phone')
    
    # ❌ 问题2：密码明文存储
    password = request.data.get('password')
    User.objects.create(phone=phone, password=password)
    
    # ❌ 问题3：没有检查是否已注册
    return JsonResponse({'status': 'ok'})
```

**攻击者可以这样利用**：

```bash
# 1. 注册大量无效手机号
curl -X POST http://example.com/register \
  -d '{"phone": "123", "password": "123"}'

# 2. 密码明文存储 → 数据库泄露后，所有用户密码暴露

# 3. 重复注册没有提示 → 可以通过重复注册探测已注册的手机号
```

---

### 🎯 正确的注册流程应该是什么样的？

```
用户提交注册
    ↓
1️⃣ 数据验证（手机号格式、密码强度）
    ↓
2️⃣ 业务验证（是否已注册、验证码是否正确）
    ↓
3️⃣ 密码加密（BCrypt加密，不可逆）
    ↓
4️⃣ 数据持久化（保存到数据库）
    ↓
5️⃣ 返回响应（隐藏敏感信息）
```

**为什么需要这么多层？**

```
没有分层：
Controller直接操作数据库
→ 代码混乱
→ 难以测试
→ 难以维护

有分层：
Controller → 接收请求
    ↓
Service    → 业务逻辑
    ↓
Repository → 数据访问
→ 职责清晰
→ 易于测试
→ 便于维护
```

### 5.1 设计用户模型

#### 🤔 为什么不能像Django那样直接继承User？

**Django的方式**：

```python
from django.contrib.auth.models import AbstractUser

class User(AbstractUser):
    phone = models.CharField(max_length=11, unique=True)
    nickname = models.CharField(max_length=50)

# 继承AbstractUser，直接获得：
# - username
# - password (自动加密)
# - email
# - is_staff
# - 等等...
```

**Java的问题**：

```java
// Java没有这样的"基类"可以直接继承
// 为什么？
// 1. Java的认证体系不是强制的
// 2. 不同项目有不同的用户字段
// 3. Spring Security的User接口太复杂

// 所以：需要自己定义用户实体
@Entity
@Table(name = "user")  # user是MySQL关键字，需要加``引号
@Data
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(unique = true, length = 11, nullable = false)
    private String phone;

    @Column(nullable = false, length = 100)
    private String password;  # 加密后的密码

    @Column(length = 50)
    private String nickname;

    // ... 更多字段
}
```

**为什么Java要这样设计？**

| 特性 | Django | Java | 原因 |
|------|--------|-----|------|
| **用户模型** | 内置AbstractUser | 自己定义 | 灵活性 |
| **密码加密** | 自动加密 | 手动加密 | 明确控制 |
| **认证方式** | Session为主 | JWT为主 | 架构差异 |

---

#### 📊 字段设计详解

**为什么手机号要设置`length=11`和`unique=true`？**

```java
@Column(unique = true, length = 11, nullable = false)
private String phone;
```

| 约束 | 作用 | 不加会怎样 |
|------|------|----------|
| `length = 11` | 限制字符串长度 | 数据库可能分配过多空间 |
| `unique = true` | 不允许重复 | 两个人用同一手机号注册！ |
| `nullable = false` | 不能为空 | 注册时没有手机号 |

**数据库层面的影响**：

```sql
-- Java的注解会生成这样的SQL
CREATE TABLE user (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    phone VARCHAR(11) NOT NULL UNIQUE,  -- length, nullable, unique
    password VARCHAR(100) NOT NULL,
    nickname VARCHAR(50),
    created_at DATETIME,
    updated_at DATETIME,
    UNIQUE KEY uk_phone (phone)  -- 唯一索引
);
```

**为什么密码字段是100字符？**

```
BCrypt加密后的密码格式：
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy

// 长度固定在60字符左右
// 为什么设置100？
// 1. 为未来加密算法升级留空间
// 2. 避免未来迁移时字段长度不够
```

### 5.2 创建Repository

#### 🤔 为什么需要Repository这一层？

**小王的疑惑**：不能直接在Controller里写数据库操作吗？

---

**场景：直接在Controller操作数据库（反面教材）**

```python
# Django反面教材
def register(request):
    # ❌ 直接在视图里操作数据库
    user = User.objects.create(
        phone=request.data['phone'],
        password=request.data['password']
    )
    return JsonResponse({'status': 'ok'})
```

```java
// Java反面教材
@RestController
public class UserController {

    @Autowired
    private UserRepository userRepository;  // ❌ Controller直接用Repository

    @PostMapping("/register")
    public Result<User> register(@RequestBody UserRegisterRequest request) {
        User user = new User();
        user.setPhone(request.getPhone());
        user.setPassword(request.getPassword());
        // ❌ Controller直接操作数据库
        User saved = userRepository.save(user);
        return Result.success(saved);
    }
}
```

**问题**：

```
1. Controller职责不清晰（既要处理请求，又要处理业务）
2. 代码复用困难（如果有多个地方需要注册逻辑呢？）
3. 测试困难（测试Controller时必须连数据库）
4. 事务管理混乱（什么时候开启事务？）
```

---

#### ✅ 正确做法：Service层分离

```java
// Controller：只负责接收请求、返回响应
@RestController
@RequestMapping("/api/user")
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;  // 注入Service

    @PostMapping("/register")
    public Result<UserResponse> register(@Valid @RequestBody UserRegisterRequest request) {
        // 调用Service处理业务逻辑
        User user = userService.register(request);
        return Result.success(UserResponse.fromEntity(user));
    }
}

// Service：负责业务逻辑
@Service
@RequiredArgsConstructor
public class UserService {

    private final UserRepository userRepository;
    private final PasswordEncoder passwordEncoder;

    @Transactional(rollbackFor = Exception.class)  // 事务管理
    public User register(UserRegisterRequest request) {
        // 1. 业务验证
        if (userRepository.existsByPhone(request.getPhone())) {
            throw new BusinessException("手机号已注册");
        }

        // 2. 密码加密
        String encodedPassword = passwordEncoder.encode(request.getPassword());

        // 3. 创建用户
        User user = new User();
        user.setPhone(request.getPhone());
        user.setPassword(encodedPassword);

        // 4. 保存到数据库
        return userRepository.save(user);
    }
}

// Repository：只负责数据访问
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByPhone(String phone);
    boolean existsByPhone(String phone);
}
```

**分层的好处**：

| 层级 | 职责 | 可以做什么 | 不能做什么 |
|------|------|-----------|-----------|
| **Controller** | 接收请求、返回响应 | 验证参数格式 | 不能有业务逻辑 |
| **Service** | 业务逻辑、事务控制 | 调用Repository | 不能直接处理请求/响应 |
| **Repository** | 数据访问 | 增删改查 | 不能有业务逻辑 |

---

#### 🎯 Repository的魔法：方法名即查询

**Django的QuerySet**：

```python
# Django：查询用方法
User.objects.filter(phone='13800138000')
User.objects.get(id=1)
User.objects.filter(nickname__contains='王')
```

**Spring Data JPA的方法名查询**：

```java
// Spring Boot：方法名就是查询
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    // findBy + 字段名 = 查询
    User findByPhone(String phone);

    // findBy + 字段名 + Containing = 模糊查询
    List<User> findByNicknameContaining(String keyword);

    // exists + By + 字段名 = 检查是否存在
    boolean existsByPhone(String phone);

    // Spring会自动实现这些方法！
    // 不需要写SQL！
}
```

**原理**：

```
Spring Data JPA解析方法名：
findByNicknameContaining
  ↓
解析为JPQL：
SELECT u FROM User u WHERE u.nickname LIKE :keyword
  ↓
执行查询，返回结果
```

**为什么这样设计？**

1. **类型安全**：编译时就能发现错误
2. **IDE支持**：IDEA会自动提示方法名
3. **可读性强**：看方法名就知道查询什么

### 5.3 创建DTO（数据传输对象）

#### 🤔 为什么不能直接用Entity作为接口返回？

**场景：直接返回Entity的安全隐患**

```python
# Django反面教材
def register(request):
    user = User.objects.create(...)
    serializer = UserSerializer(user)
    return Response(serializer.data)
    # 问题：密码会被返回！
```

```java
// Java反面教材
@PostMapping("/register")
public Result<User> register(@RequestBody UserRegisterRequest request) {
    User user = new User();
    user.setPassword(passwordEncoder.encode(request.getPassword()));
    user = userRepository.save(user);
    return Result.success(user);  // ❌ 密码被返回了！
}
```

**攻击者可以看到**：

```json
{
  "code": 200,
  "data": {
    "id": 1,
    "phone": "13800138000",
    "password": "$2a$10$N9qo8uLOickgx2ZMRZoMye",  // ❌ 加密后的密码
    "nickname": "小王"
  }
}
```

即使密码加密了，也不应该返回！

---

#### ✅ 正确做法：DTO分离

**请求DTO（接收用户数据）**：

```java
@Data
public class UserRegisterRequest {
    @NotBlank(message = "手机号不能为空")
    @Pattern(regexp = "^1\\d{10}$", message = "手机号格式错误")
    private String phone;

    @NotBlank(message = "密码不能为空")
    @Size(min = 6, max = 20, message = "密码长度6-20位")
    private String password;

    @NotBlank(message = "验证码不能为空")
    private String captcha;

    @Size(max = 50, message = "昵称最长50位")
    private String nickname;

    // 注意：不包含敏感字段（如id、created_at）
}
```

**响应DTO（返回给前端）**：

```java
@Data
public class UserResponse {
    private Long id;
    private String phone;
    private String nickname;
    private String avatar;
    private LocalDateTime createdAt;

    // 注意：不包含password等敏感信息

    // Entity → DTO 转换方法
    public static UserResponse fromEntity(User user) {
        UserResponse response = new UserResponse();
        response.setId(user.getId());
        response.setPhone(user.getPhone());
        response.setNickname(user.getNickname());
        response.setAvatar(user.getAvatar());
        response.setCreatedAt(user.getCreatedAt());
        // 故意不复制password字段
        return response;
    }
}
```

**使用DTO的好处**：

| 好处 | 说明 |
|------|------|
| **安全性** | 不暴露敏感信息 |
| **灵活性** | 前后端数据格式可以不同 |
| **验证分离** | 请求验证和响应验证分开 |
| **版本控制** | API v1和v2可以用不同的DTO |

### 5.4 密码加密

#### 🤔 为什么不能明文存储密码？

**真实案例：2012年LinkedIn密码泄露事件**

```
泄露的密码：
- 明文存储的密码直接暴露
- 6.5万个用户密码被破解
- 很多用户在多个网站用相同密码
```

**为什么不存储明文密码？**

1. **数据库泄露**：管理员恶意、SQL注入、备份被盗
2. **内部人员**：DBA、运维可以直接看到数据库
3. **日志泄露**：应用日志可能记录密码
4. **社会工程学**：用户的密码可能是其他网站的密码

---

#### 🔐 BCrypt加密算法

**Django的方式**：

```python
from django.contrib.auth.hashers import make_password, check_password

# 加密
password = make_password('123456')
# 结果：$2b$12$QjRjz...（60字符）

# 验证
if check_password('123456', password):
    print('密码正确')
```

**Spring Boot的方式**：

```java
// 配置Bean
@Configuration
public class SecurityConfig {

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}

// 使用
@Service
@RequiredArgsConstructor
public class UserService {

    private final PasswordEncoder passwordEncoder;

    public User register(UserRegisterRequest request) {
        // 加密密码
        String encodedPassword = passwordEncoder.encode(request.getPassword());
        user.setPassword(encodedPassword);

        // 验证密码
        if (passwordEncoder.matches(request.getPassword(), user.getPassword())) {
            // 密码正确
        }
    }
}
```

**为什么选择BCrypt？**

| 算法 | 优点 | 缺点 |
|------|------|------|
| **MD5/SHA-1** | 快速 | 已经被破解，不安全 |
| **SHA-256/512** | 比MD5安全 | 可以用彩虹表攻击 |
| **BCrypt** | 自动加盐、慢（防暴力破解） | 比较慢 |
| **Argon2** | BCrypt的升级版 | 较新，兼容性稍差 |

**BCrypt的优势**：

```
1. 自动加盐：每次加密结果都不同
   '123456' → $2a$10$abc...
   '123456' → $2a$10$def...

2. 计算慢：防止暴力破解
   MD5：每秒百万次
   BCrypt：每秒几千次

3. 可调整强度：cost参数控制计算复杂度
```

### 5.5 参数验证

#### 🤔 为什么需要这么多验证注解？

**场景：没有验证的后果**

```python
# 没有验证
def register(request):
    phone = request.data.get('phone')
    # 用户可能传入：
    # - 空字符串
    # - 任意字符
    # - 超长字符串
    # - null
```

**Java的验证注解**：

```java
@Data
public class UserRegisterRequest {

    @NotBlank(message = "手机号不能为空")
    @Pattern(regexp = "^1\\d{10}$", message = "手机号格式错误")
    private String phone;

    @NotBlank(message = "密码不能为空")
    @Size(min = 6, max = 20, message = "密码长度6-20位")
    private String password;

    @NotBlank(message = "验证码不能为空")
    private String captcha;
}
```

**验证过程**：

```
1. 用户提交数据
   ↓
2. @Valid注解触发验证
   ↓
3. 检查@NotBlank → phone不能为空
   ↓
4. 检查@Pattern → phone必须匹配正则
   ↓
5. 检查@Size → 密码长度必须在6-20
   ↓
6. 验证通过 → 进入Controller
   或
   验证失败 → 抛出异常 → GlobalExceptionHandler捕获
```

**验证失败的响应**：

```json
{
  "code": 400,
  "message": "phone: 手机号格式错误, password: 密码长度6-20位"
}
```

### 5.6 全局异常处理

#### 🤔 为什么需要全局异常处理？

**场景：没有统一异常处理的混乱**

```python
# 每个方法都要处理异常
def register(request):
    try:
        user = create_user(request.data)
    except ValidationError as e:
        return Response({'error': str(e)}, status=400)
    except IntegrityError as e:
        return Response({'error': '手机号已注册'}, status=400)
    except Exception as e:
        return Response({'error': '系统错误'}, status=500)

def login(request):
    try:
        user = authenticate(request.data)
    except ValidationError as e:
        return Response({'error': str(e)}, status=400)
    except AuthenticationFailed as e:
        return Response({'error': '用户名或密码错误'}, status=401)
    except Exception as e:
        return Response({'error': '系统错误'}, status=500)
# ... 每个方法都重复这些代码
```

**全局异常处理**：

```java
@RestControllerAdvice  // 拦截所有Controller的异常
public class GlobalExceptionHandler {

    // 处理参数验证异常
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<?> handleValidException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors()
            .stream()
            .map(error -> error.getField() + ": " + error.getDefaultMessage())
            .collect(Collectors.joining(", "));
        return Result.badRequest(message);
    }

    // 处理业务异常
    @ExceptionHandler(BusinessException.class)
    public Result<?> handleBusinessException(BusinessException e) {
        return Result.error(e.getMessage());
    }

    // 处理所有异常（兜底）
    @ExceptionHandler(Exception.class)
    public Result<?> handleException(Exception e) {
        e.printStackTrace();  // 开发环境打印堆栈
        return Result.error("系统繁忙，请稍后重试");
    }
}

// 现在Controller不需要try-catch了！
@PostMapping("/register")
public Result<UserResponse> register(@Valid @RequestBody UserRegisterRequest request) {
    // 不需要try-catch，异常会被全局处理器捕获
    User user = userService.register(request);
    return Result.success(UserResponse.fromEntity(user));
}
```

### 5.7 完整注册流程

#### 📋 端到端流程

```
┌─────────────────────────────────────────────────────┐
│                      用户                              │
└──────────────────────┬──────────────────────────────┘
                       │
                       ↓
┌─────────────────────────────────────────────────────┐
│              前端（React/Vue）                        │
│  POST /api/user/register                           │
│  {                                                  │
│    "phone": "13800138000",                         │
│    "password": "123456",                            │
│    "captcha": "1234"                                │
│  }                                                  │
└──────────────────────┬──────────────────────────────┘
                       │
                       ↓
┌─────────────────────────────────────────────────────┐
│              Controller层                            │
│  - @Valid 触发参数验证                              │
│  - 格式正确 → 进入Service                            │
│  - 格式错误 → 返回400                               │
└──────────────────────┬──────────────────────────────┘
                       │
                       ↓
┌─────────────────────────────────────────────────────┐
│              Service层                               │
│  1. 检查手机号是否已注册                            │
│  2. 验证验证码                                       │
│  3. 密码加密                                         │
│  4. 创建User对象                                    │
│  - @Transactional 保证事务                           │
└──────────────────────┬──────────────────────────────┘
                       │
                       ↓
┌─────────────────────────────────────────────────────┐
│              Repository层                            │
│  - save(user) 保存到数据库                          │
│  - Spring Data JPA自动生成SQL                       │
└──────────────────────┬──────────────────────────────┘
                       │
                       ↓
┌─────────────────────────────────────────────────────┐
│              MySQL数据库                             │
│  INSERT INTO user (phone, password, ...) VALUES (...)  │
└──────────────────────┬──────────────────────────────┘
                       │
                       ↓
              返回用户信息（不含密码）
```

### 5.8 本章小结

#### 🎯 核心要点

| 概念 | 为什么要这样设计 |
|------|----------------|
| **分层架构** | 职责清晰，便于维护和测试 |
| **Repository** | 封装数据访问，支持方法名查询 |
| **DTO** | 前后端数据分离，保护敏感信息 |
| **密码加密** | BCrypt保证密码安全 |
| **参数验证** | @Valid注解自动验证，减少手动检查 |
| **全局异常** | 统一处理错误，避免重复代码 |

#### 💡 常见错误

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| `NullPointerException` | 忘记判空 | 使用Optional或if检查 |
| `DataIntegrityViolationException` | 唯一约束冲突 | 先检查再插入 |
| `BindException` | 参数验证失败 | 检查字段格式 |
| `TransactionException` | 事务回滚 | 检查数据库连接 |
