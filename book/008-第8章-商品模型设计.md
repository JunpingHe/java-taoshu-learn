## 🤔 为什么商品模型这么复杂？

> **小王的疑惑**：不就是一本书吗？书名、作者、价格，不就完了吗？为什么需要这么多字段？

### 从一个真实需求说起

**场景1：用户想买二手书，需要什么信息？**

```
买书人关心：
✅ 书名、作者 - 是不是我想要的书？
✅ ISBN - 确认是哪一版
✅ 价格 - 便宜吗？
✅ 成色 - 新书还是二手？
✅ 卖家 - 卖家信誉如何？
✅ 库存 - 还有货吗？
✅ 图片 - 书长什么样？

如果缺少这些信息：
→ 用户不敢买
→ 卖家要反复回答
→ 成交效率低
```

**场景2：卖家想卖书，需要记录什么信息？**

```
卖书人关心：
✅ 书籍基本信息 - 让买家了解
✅ 定价策略 - 原价多少，卖多少？
✅ 库存管理 - 还有几本可卖？
✅ 销售数据 - 卖了多少本？
✅ 成色说明 - 新旧程度
✅ 发布时间 - 上架多久了

如果缺少这些信息：
→ 卖家无法管理库存
→ 定价没有依据
→ 无法分析销售数据
```

**场景3：平台需要什么信息？**

```
平台关心：
✅ 搜索优化 - 关键词字段
✅ 分类管理 - 图书归类
✅ 推荐算法 - 热销、新书
✅ 数据统计 - 交易额、活跃度
✅ 信用管理 - 卖家信誉、买家信誉
✅ 风控管理 - 违禁商品、禁用用户
✅ 运营分析 - 哪类偏好、价格区间

如果缺少这些信息：
→ 搜索功能不好用
→ 无法做推荐
→ 无法做数据分析
```

### 🎯 完整的Book实体设计

#### 为什么需要这么多字段？

```java
@Entity
@Table(name = "book")
@Data
public class Book {

    // ============ 基本信息 ============
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 200)
    private String title;  // 书名（必填，用于搜索）

    @Column(length = 100)
    private String author;  // 作者（用于搜索、展示）

    @Column(length = 20, unique = true)
    private String isbn;  // ISBN（唯一标识，防止重复）

    @Column(length = 100)
    private String publisher;  // 出版社

    // ============ 分类信息 ============
    @Column(name = "category_id")
    private Long categoryId;  // 分类ID（外键）

    @Column(name = "category_name", length = 50)
    private String categoryName;  // 冗余分类名（避免JOIN查询）

    // 为什么冗余分类名？
    // 如果不冗余，每次显示都要JOIN category表
    // 冗余字段用空间换时间

    // ============ 价格信息 ============
    @Column(nullable = false, precision = 10, scale = 2)
    private BigDecimal price;  // 现价

    @Column(name = "original_price", precision = 10, scale = 2)
    private BigDecimal originalPrice;  // 原价

    // 为什么需要原价？
    // → 计算折扣：3折、5折
    // → 营销效果：显示"原价100元，现价30元"
    // → 心理暗示：感觉占了便宜

    // ============ 库存信息 ============
    @Column(name = "stock_quantity", nullable = false)
    private Integer stockQuantity = 0;  // 库存数量

    @Column(name = "sold_quantity", nullable = false)
    private Integer soldQuantity = 0;  // 已售数量

    // 为什么需要这两个字段？
    // → stockQuantity：防止超卖
    // → soldQuantity：显示"已售XX本"，增加购买紧迫感

    // ============ 状态信息 ============
    @Column(columnDefinition = "TINYINT DEFAULT 1")
    private Integer status = 1;  // 0-下架, 1-上架

    @Column(columnDefinition = "TINYINT DEFAULT 1")
    private Integer condition = 1;  // 1-全新, 2-九成新...

    // ============ 卖家信息 ============
    @Column(name = "seller_id", nullable = false)
    private Long sellerId;  // 卖家ID

    @Column(name = "seller_name", length = 50)
    private String sellerName;  // 冗余卖家名

    // ============ 图片描述 ============
    @Column(name = "cover_image", columnDefinition = "TEXT")
    private String coverImage;  // 封面图URL

    @Column(columnDefinition = "TEXT")
    private String description;  // 内容简介

    @Column(name = "detail", columnDefinition = "TEXT")
    private String detail;  // 详情（富文本）

    // ============ 时间戳 ============
    @CreatedDate
    @Column(name = "created_at", updatable = false)
    private LocalDateTime createdAt;  // 创建时间

    @LastModifiedDate
    @Column(name = "updated_at")
    private LocalDateTime updatedAt;  // 更新时间

    @Column(name = "is_deleted", columnDefinition = "TPIYINT DEFAULT 0")
    private Boolean isDeleted = false;  // 软删除
}
```

### 8.1 冗余字段：空间换时间

#### 🤔 为什么要在Book表存category_name？

**问题：不冗余会怎样？**

```
不冗余设计：
book表：
  - id
  - title
  - category_id (外键)

category表：
  - id
  - name

查询图书列表时：
SELECT b.*, c.name
FROM book b
LEFT JOIN category c ON b.category_id = c.id
→ 每次都要JOIN

冗余设计：
book表：
  - id
  - title
  - category_id
  - category_name (冗余)

查询图书列表时：
SELECT b.*
FROM book b
→ 不需要JOIN
→ 性能更好
```

**对比**：

| 方面 | 不冗余（JOIN） | 冗余字段 |
|------|---------------|---------|
| **查询性能** | 需要JOIN，慢 | 直接查，快 |
| **存储空间** | 省空间 | 多一些字段 |
| **数据一致性** | 修改方便 | 需要同步更新 |
| **业务场景** | 分类经常变 | 分类基本不变 |

**选择冗余的场景**：

```
✅ 适合冗余：
- 读多写少（书籍查询多，分类修改少）
- 分类字段相对稳定
- 需要提升查询性能

❌ 不适合冗余：
- 分类经常修改
- 存储空间紧张
- 数据一致性要求高
```

### 8.2 价格字段设计

#### 🤔 为什么需要原价字段？

**场景：促销策略**

```
不打折：
书名：Java编程思想
现价：88元
→ 用户不知道贵不贵

显示原价：
书名：Java编程思想
原价：128元  现价：88元 → 7折
→ 用户知道打了7折
→ 感觉占便宜

折扣计算：
折扣 = 现价 / 原价 × 10
88 / 128 × 10 = 6.875 ≈ 7折
```

**实现**：

```java
@Data
public class BookResponse {

    private BigDecimal price;
    private BigDecimal originalPrice;
    private String discount;  // 计算属性

    public static BookResponse fromEntity(Book book) {
        BookResponse response = new BookResponse();
        response.setPrice(book.getPrice());
        response.setOriginalPrice(book.getOriginalPrice());

        // 计算折扣
        if (book.getOriginalPrice() != null &&
            book.getOriginalPrice().compareTo(BigDecimal.ZERO) > 0) {

            BigDecimal discount = book.getPrice()
                .divide(book.getOriginalPrice(), 2, RoundingMode.HALF_UP)
                .multiply(BigDecimal.valueOf(10));

            response.setDiscount(discent.toString() + "折");
        }

        return response;
    }
}
```

### 8.3 库存字段设计

#### 🤔 为什么需要stock_quantity和sold_quantity两个字段？

**场景1：库存管理**

```
只有stockQuantity：
→ 不知道卖了多少
→ 无法分析销售速度
→ 不知道什么时候补货

只有soldQuantity：
→ 不知道还剩多少
→ 无法判断是否超卖
→ 无法展示库存状态
```

**场景2：购买流程**

```java
// 购买前检查
if (book.getStockQuantity() < quantity) {
    throw new BusinessException("库存不足");
}

// 扣减库存
book.setStockQuantity(book.getStockQuantity() - quantity);

// 增加已售
book.setSoldQuantity(book.getSoldQuantity() + quantity);

// 展示给用户
库存：还有XX本可购
已售：已售XX本（增加紧迫感）
```

**为什么不用事务扣减？**

```
方案A：下单时扣减库存
✅ 优点：实时准确
❌ 缺点：下单不付款也扣库存

方案B：支付时扣减库存
✅ 优点：付款才扣库存
❌ 缺点：可能超卖（需要锁）

方案C：预占库存
✅ 优点：防止超卖
❌ 缺点：实现复杂

我们选择：下单时扣减 + 超时未付款取消
```

### 8.4 成色字段设计

#### 🤔 为什么要有成色这个概念？

**场景：二手书交易的特殊性**

```
全新书：
→ 可以当新书卖
→ 价格接近新书
→ 不需要特殊说明

二手书：
→ 有使用痕迹
→ 价格差异大
→ 需要说明成色

没有成色字段：
→ "这本书8成新"
→ 什么算8成新？
→ 靠人工判断，标准不统一
→ 容易产生纠纷

有成色字段：
→ 标准化的成色等级
→ 1-全新，2-九成新，3-八成新...
→ 买家一目了然
→ 减少纠纷
```

**成色等级定义**：

```java
public enum BookCondition {
    BRAND_NEW(1, "全新"),           // 刚拆封，未使用
    LIKE_NEW(2, "九成新"),           # 使用痕迹很少
    GOOD(3, "八成新"),             # 有明显使用痕迹
    ACCEPTABLE(4, "七成新"),       # 使用较多，有磨损
    WORN(5, "六成新及以下");      # 使用较多，有明显磨损

    private final Integer code;
    private final String description;

    // 使用枚举而不是魔法数字
    @Column(columnDefinition = "TINYINT DEFAULT 1")
    private Integer condition = 1;
}
```

**前端展示**：

```javascript
// React组件
const BookDetail = ({ book }) => {
```

### 8.5 状态字段设计

#### 🤔 为什么需要状态管理？

**场景：图书的生命周期**

```
1. 草稿状态（未上架）
   → 卖家还在编辑信息
   → 还没准备好出售

2. 上架状态（status=1）
   → 可以被买家看到
   → 可以下单购买

3. 下架状态（status=0）
   → 暂时不想卖了
   → 图书信息还在，但不能买
   → 随时可以重新上架

4. 已售出状态
   → 书已经卖给别人
   → 不再展示
   → 从搜索结果中移除

5. 软删除
   → 彻底删除
   → 数据库中移除
```

**状态转换**：

```
上架(1) ←→ 下架(0)
    ↓
已售出(2)
    ↓
删除
```

**实现**：

```java
@Service
@RequiredArgsConstructor
public class BookService {

    // 下架图书
    public void takeOffBook(Long bookId, Long sellerId) {
        Book book = getById(bookId);

        // 验证权限
        if (!book.getSellerId().equals(sellerId)) {
            throw new BusinessException("无权操作");
        }

        // 下架
        book.setStatus(0);
        bookRepository.save(book);
    }

    // 上架图书
    public void listBook(Long bookId, Long sellerId) {
        Book book = getById(bookId);

        if (!book.getSellerId().equals(sellerId)) {
            throw new BusinessException("无权操作");
        }

        book.setStatus(1);
        bookRepository.save(book);
    }

    // 标记为已售出
    @Transactional
    public void markAsSold(Long bookId) {
        Book book = getById(bookId);
        book.setStatus(2);  // 已售出
        bookRepository.save(book);
    }
}
```

### 8.6 卖家信息字段

#### 🤔 为什么要在Book表存seller_id？

**问题：不存卖家ID会怎样？**

```
不存seller_id：
→ 查询某卖家的书？
→ 需要JOIN user表
→ 性能差

存seller_id：
→ 直接WHERE seller_id = ?
→ 性能好

冗余seller_name：
→ 显示时不需要JOIN
→ 但改名需要同步
```

**对比**：

```sql
-- 不冗余
SELECT b.*, u.username as seller_name
FROM book b
LEFT JOIN user u ON b.seller_id = u.id
WHERE b.seller_id = 123

-- 冗余
SELECT b.*
FROM book b
WHERE b.seller_id = 123
seller_name直接取
```

**选择**：读写比1:100，适合冗余

### 8.7 软删除 vs 硬删除

#### 🤔 为什么要用软删除？

**硬删除（DELETE）的问题**：

```
问题1：数据无法恢复
→ 删除错了
→ 无法找回
→ 数据永久丢失

问题2：关联数据丢失
→ 订单表有book_id外键
→ 图书被删除后
→ 订单记录不完整

问题3：无法分析
→ 无法统计历史销售
→ 无法分析删除趋势
```

**软删除解决方案**：

```java
@Column(name = "is_deleted", columnDefinition = "TINYINT DEFAULT 0")
private Boolean isDeleted = false;

// 查询时自动过滤
@Query("SELECT b FROM Book b WHERE b.isDeleted = false")
List<Book> findAll();

// 删除时只标记
public void deleteBook(Long bookId) {
    Book book = getById(bookId);
    book.setIsDeleted(true);
    bookRepository.save(book);
}
```

**软删除的好处**：

| 好处 | 说明 |
|------|------|
| **数据恢复** | 误删可以恢复 |
| **数据完整** | 历史订单数据完整 |
| **分析统计** | 可以分析删除趋势 |
| **合规要求**| 某些行业要求保留数据 |

### 8.9 数据库索引设计

#### 🤔 为什么需要在这些字段上建索引？

**慢查询问题**：

```sql
-- 没有索引的查询
SELECT * FROM book WHERE seller_id = 123;
→ 全表扫描，慢！

-- 有索引后
SELECT * FROM book WHERE seller_id = 123;
→ 使用索引，快！
```

**需要建索引的字段**：

```java
@Entity
@Table(name = "book",
    indexes = {
        @Index(name = "idx_seller_status", columnList = "seller_id, status"),
        @Index(name = "idx_category_status", columnList = "category_id, status"),
        @Index(name = "idx_isbn", columnList = "isbn"),
        @Index(name = "idx_status_created", columnList = "status, created_at")
    }
)
public class Book {
    // ...
}
```

**为什么这些组合索引？**

| 索引 | 用途 | 示例查询 |
|------|------|----------|
| `idx_seller_status` | 查询某卖家的上架图书 | `WHERE seller_id=123 AND status=1` |
| `idx_category_status` | 查询某分类的图书 | `WHERE category_id=5 AND status=1` |
| `idx_isbn` | 根据ISBN查询 | `WHERE isbn='9787111...'` |
| `idx_status_created` | 最新上架图书 | `WHERE status=1 ORDER BY created_at DESC` |

### 8.10 本章小结

#### 🎯 Book实体设计要点

| 字段类别 | 字段 | 作用 |
|---------|------|------|
| **基本信息** | title, author, isbn, publisher | 识别、搜索 |
| **分类** | category_id, category_name | 分类、过滤 |
| **价格** | price, original_price | 展示、计算折扣 |
| **库存** | stock_quantity, sold_quantity | 库存管理、销售统计 |
| **状态** | status, condition, is_deleted | 上架状态、成色、软删除 |
| **卖家** | seller_id, seller_name | 关联卖家、权限控制 |
| **图片** | cover_image, description, detail | 展示、详情 |

#### 💡 设计原则

1. **冗余字段**：用空间换时间（如category_name）
2. **枚举类型**：用标准替代魔法数字
3. **软删除**：保留数据，可恢复
4. **索引优化**：为常用查询字段建索引
5. **完整信息**：满足买家、卖家、平台三方需求
