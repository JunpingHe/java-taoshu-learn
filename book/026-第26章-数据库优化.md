# 第二十六章：数据库优化

> **小王的疑问**：随着数据量增加，查询越来越慢...10万条数据还好，100万条明显变慢，1000万条几乎无法使用...怎么办？

---

## 🤔 为什么需要数据库优化？

### 真实场景：性能瓶颈

```
某电商平台未优化数据库：
┌─────────────────────────────────────────────────────────────┐
│  数据量：10万 → 100万 → 1000万                            │
│                                                             │
│  10万数据：                                                 │
│  - 商品列表查询：50ms                                      │
│  - 订单查询：30ms                                          │
│  - 用户体验：良好                                          │
│                                                             │
│  100万数据：                                                │
│  - 商品列表查询：500ms                                     │
│  - 订单查询：300ms                                         │
│  - 用户体验：一般                                          │
│                                                             │
│  1000万数据：                                               │
│  - 商品列表查询：5000ms（5秒！）                            │
│  - 订单查询：3000ms（3秒！）                               │
│  - 用户体验：极差 → 用户流失                              │
│  - 数据库CPU：100%                                         │
│  - 响应超时 → 应用崩溃                                     │
│                                                             │
│  优化后：                                                   │
│  - 1000万数据：查询100ms                                   │
│  - 用户体验：流畅                                          │
│  - 数据库CPU：30%                                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**数据库优化的价值**：
1. **提升性能**：查询速度提升10-100倍
2. **降低成本**：同样的硬件支撑更多用户
3. **提高稳定性**：避免慢查询导致的超时
4. **改善体验**：页面加载快，用户满意

## 第一步：识别慢查询

### 问题：如何找出慢查询？

```
慢查询定义：
执行时间 > 100ms 的查询

危害：
- 占用数据库连接
- 锁表，阻塞其他查询
- CPU/内存/IO消耗高
- 用户体验差
```

## 第二步：索引优化

### 问题：为什么需要索引？

```
类比：书本的目录
没有目录：
想找"Java语法"章节 → 一页页翻 → 很慢

有目录：
查目录 → 第156页 → 直接翻到 → 很快

数据库索引：
- 无索引：全表扫描（慢）
- 有索引：索引查找（快）

但索引不是万能的：
- 占用存储空间
- 降低写入性能
- 需要定期维护
```

### 代码实现：添加索引

```sql
-- 1. 主键索引（自动创建）
-- 已有主键：id

-- 2. 唯一索引（唯一字段）
CREATE UNIQUE INDEX uk_order_no ON `order`(order_no);

-- 3. 普通索引（经常查询的字段）
CREATE INDEX idx_user_id ON `order`(user_id);
CREATE INDEX idx_status ON `order`(status);
CREATE INDEX idx_created_at ON `order`(created_at);

-- 4. 复合索引（多字段组合查询）
CREATE INDEX idx_user_status ON `order`(user_id, status);
CREATE INDEX idx_status_created ON `order`(status, created_at);

-- 5. 覆盖索引（查询所需字段都在索引中）
CREATE INDEX idx_user_status_amount ON `order`(user_id, status, pay_amount);

-- 查看索引
SHOW INDEX FROM `order`;

-- 删除索引
DROP INDEX idx_user_id ON `order`;
```

### 索引设计原则

```
1. 选择性高的字段（唯一值多）
   ✅ order_no（唯一，选择性高）
   ✅ user_id（选择性高）
   ❌ status（只有几个值，选择性低）

2. 经常用于查询的字段
   WHERE user_id = ?
   WHERE status = ? AND created_at > ?

3. 经常用于排序的字段
   ORDER BY created_at DESC

4. 覆盖索引（回表优化）
   SELECT id FROM user WHERE user_id = ?
   # 如果索引包含id，不需要回表

5. 最左前缀原则（复合索引）
   INDEX(a, b, c)
   ✅ WHERE a = ?
   ✅ WHERE a = ? AND b = ?
   ✅ WHERE a = ? AND b = ? AND c = ?
   ❌ WHERE b = ?（跳过a，不使用索引）
   ❌ WHERE c = ?（跳过a、b，不使用索引）
```

## 第三步：SQL优化

### 问题：如何优化SQL？

#### 1. 避免 SELECT *

```java
// ❌ 不推荐
@Query("SELECT b FROM Book b")
List<Book> findAll();

// ✅ 推荐（明确需要的字段）
@Query("SELECT new com.taoshu.dto.BookSimple(b.id, b.title, b.price) FROM Book b")
List<BookSimple> findAllSimple();

// 优势：
// 1. 减少网络传输
// 2. 减少内存占用
// 3. 可能使用覆盖索引
```

#### 2. 分页查询

```java
// ❌ 不推荐：查询所有再分页
List<Book> books = bookRepository.findAll();
Page<Book> page = // 内存分页

// ✅ 推荐：数据库分页
Page<Book> books = bookRepository.findAll(PageRequest.of(0, 20));

// SQL：LIMIT 20 OFFSET 0
```

#### 3. 批量操作

```java
// ❌ 不推荐：N条SQL
for (Order order : orders) {
    orderRepository.save(order);  # N条UPDATE
}

// ✅ 推荐：批量操作
List<Order> savedOrders = orderRepository.saveAll(orders);  # 1条批量SQL
```

#### 4. JOIN优化

```sql
-- ❌ 不推荐：子查询
SELECT * FROM order WHERE user_id IN (SELECT user_id FROM user WHERE status = 1);

-- ✅ 推荐：JOIN
SELECT o.* FROM `order` o
INNER JOIN user u ON o.user_id = u.id
WHERE u.status = 1;
```

## 第四步：连接池优化

### 问题：连接数设置多少合适？

```
连接池的作用：
- 复用数据库连接
- 限制最大连接数
- 管理连接生命周期

参数设置：
minimum-idle：最小空闲连接
maximum-pool-size：最大连接数
connection-timeout：连接超时
idle-timeout：空闲连接超时
max-lifetime：连接最大存活时间
```

### 代码实现：HikariCP配置

```yaml
# application.yml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/taoshu?useSSL=false&serverTimezone=Asia/Shanghai
    username: root
    password: yourpassword
    driver-class-name: com.mysql.cj.jdbc.Driver
    hikari:
      # 连接池配置
      minimum-idle: 5           # 最小空闲连接
      maximum-pool-size: 20      # 最大连接数
      connection-timeout: 30000  # 连接超时（30秒）
      idle-timeout: 600000       # 空闲超时（10分钟）
      max-lifetime: 1800000      # 最大存活时间（30分钟）

      # 性能配置
      connection-test-query: SELECT 1  # 连接测试查询
      validation-timeout: 3000   # 验证超时
      leak-detection-threshold: 60000  # 连接泄露检测（60秒）

      # 其他配置
      pool-name: TaoshuHikariCP
      auto-commit: true
      read-only: false
```

**连接数计算公式**

```
公式：connections = ((core_count * 2) + effective_spindle_count)

示例：
CPU核数：8
有效线程数：100
connections = (8 * 2) + 100 = 116

实际配置：
- 开发环境：5-10
- 测试环境：10-20
- 生产环境：20-50（根据压力测试调整）
```

## 第五步：读写分离

### 问题：如何分担数据库压力？

```
场景：
读多写少（电商场景）
- 80%是读操作（查询商品、订单等）
- 20%是写操作（创建订单、更新库存等）

问题：
单库同时承担读写
→ 读压力大，影响写操作
→ 整体性能下降

解决：读写分离
- 主库（Master）：处理写操作
- 从库（Slave）：处理读操作
```

### 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                    读写分离架构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [应用]                                                     │
│      ├─ 写操作 → [主库]（Master）                           │
│      │              ↓ 同步                                 │
│      │              [从库1]（Slave） ← ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─│
│      └─ 读操作 → [从库2]（Slave） ← ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─│
│                     ↑                                      │
│                     └── 负载均衡                             │
│                                                             │
│  同步方式：                                                 │
│  1. 主库变更记录到binlog                                   │
│  2. 从库IO线程读取binlog                                   │
│  3. 从库SQL线程执行binlog                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 代码实现：动态数据源

```java
// config/DataSourceConfig.java
package com.taoshu.config;

import com.zaxxer.hikari.HikariDataSource;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.boot.jdbc.DataSourceBuilder;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Primary;
import javax.sql.DataSource;

@Configuration
public class DataSourceConfig {

    @Bean
    @ConfigurationProperties(prefix = "spring.datasource.master")
    public DataSource masterDataSource() {
        return DataSourceBuilder.create().type(HikariDataSource.class).build();
    }

    @Bean
    @ConfigurationProperties(prefix = "spring.datasource.slave")
    public DataSource slaveDataSource() {
        return DataSourceBuilder.create().type(HikariDataSource.class).build();
    }

    @Bean
    @Primary
    public DataSource routingDataSource(
            @Qualifier("masterDataSource") DataSource masterDataSource,
            @Qualifier("slaveDataSource") DataSource slaveDataSource) {

        DynamicDataSource routingDataSource = new DynamicDataSource();
        Map<Object, Object> dataSourceMap = new HashMap<>();
        dataSourceMap.put("master", masterDataSource);
        dataSourceMap.put("slave", slaveDataSource);

        routingDataSource.setTargetDataSources(dataSourceMap);
        routingDataSource.setDefaultTargetDataSource(masterDataSource);  # 默认主库

        return routingDataSource;
    }
}
```

```java
// config/DynamicDataSource.java
public class DynamicDataSource extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        // 从ThreadLocal获取数据源类型
        return DataSourceContextHolder.getDataSourceType();
    }
}
```

```java
// util/DataSourceContextHolder.java
public class DataSourceContextHolder {

    private static final ThreadLocal<String> contextHolder = new ThreadLocal<>();

    public static void setMaster() {
        contextHolder.set("master");
    }

    public static void setSlave() {
        contextHolder.set("slave");
    }

    public static String getDataSourceType() {
        return contextHolder.get();
    }

    public static void clearDataSource() {
        contextHolder.remove();
    }
}
```

**使用注解**

```java
// annotation/ReadOnly.java
@Target({ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
public @interface ReadOnly {
}

// aop/ReadOnlyAspect.java
@Aspect
@Component
public class ReadOnlyAspect {

    @Around("@annotation(ReadOnly)")
    public Object around(ProceedingJoinPoint point) throws Throwable {
        try {
            // 设置使用从库
            DataSourceContextHolder.setSlave();
            return point.proceed();
        } finally {
            // 清除
            DataSourceContextHolder.clearDataSource();
        }
    }
}

// 使用
@Service
public class OrderService {

    @ReadOnly  # 使用从库
    public Order getOrder(Long id) {
        return orderRepository.findById(id).orElse(null);
    }

    @Transactional  # 写操作，使用主库
    public Order createOrder(CreateOrderRequest request) {
        # ...
    }
}
```

## 第六步：分表策略

### 问题：单表数据量太大怎么办？

```
场景：
order表数据量：1000万+ → 查询慢

方案1：垂直分表（按字段拆分）
order表 → order_base（基础信息）+ order_detail（详情信息）
order_base：id, order_no, user_id, status, created_at
order_detail：id, goods_info, logistics_info

方案2：水平分表（按数据量拆分）
order表 → order_0, order_1, order_2, ..., order_9
按user_id取模：user_id % 10 → 分表
```

### 代码实现：ShardingSphere

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.apache.shardingsphere</groupId>
    <artifactId>shardingsphere-jdbc-core-spring-boot-starter</artifactId>
    <version>5.3.2</version>
</dependency>
```

```yaml
# application.yml
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://localhost:3306/taoshu_0
      ds1:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://localhost:3306/taoshu_1

    rules:
      sharding:
        tables:
          # order表分表
          order:
            actual-data-nodes: ds0.order_$->{0..9}, ds1.order_$->{0..9}
            table-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: order-mod
        sharding-algorithms:
          order-mod:
            type: MOD
            props:
              sharding-count: 10

    props:
      sql-show: true
```

## ⚠️ 数据库优化清单

### 优化前检查

```
□ 慢查询日志：找出慢查询

```
