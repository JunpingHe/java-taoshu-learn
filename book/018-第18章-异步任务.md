# 第十八章：异步任务

> **小王的疑问**：用户下单后需要发送短信、邮件、通知商家...这些操作都很慢，会影响下单速度吗？Django用Celery，Spring Boot用什么？

---

## 🤔 为什么需要异步任务？

### 真实场景：同步处理的灾难

```
某电商平台同步处理下单：
┌─────────────────────────────────────────────────────────────┐
│  用户点击"立即购买"                                         │
│      ↓                                                      │
│  1. 创建订单（100ms）                                       │
│  2. 扣减库存（100ms）                                       │
│  3. 发送短信（500ms） ← 慢！                                │
│  4. 发送邮件（800ms） ← 慢！                                │
│  5. 通知商家（500ms） ← 慢！                                │
│  6. 生成发票（300ms） ← 慢！                                │
│      ↓                                                      │
│  总耗时：2300ms = 2.3秒！                                   │
│                                                             │
│  用户感受：                                                 │
│  - 页面转圈2秒多                                            │
│  - 以为卡死了，重复点击                                     │
│  - 体验极差                                                 │
│                                                             │
│  系统影响：                                                 │
│  - 响应慢，用户流失                                         │
│  - 并发能力差（一个请求占用2秒）                            │
│  - 资源浪费（等待短信、邮件）                               │
└─────────────────────────────────────────────────────────────┘
```

**异步处理的价值**：
1. **提升响应速度**：主流程100ms返回，异步任务后台执行
2. **提高并发能力**：线程释放，处理更多请求
3. **解耦业务逻辑**：短信失败不影响下单
4. **提高可靠性**：异步任务失败可重试

## 第一步：理解异步模型

### 同步 vs 异步

```
┌─────────────────────────────────────────────────────────────┐
│  同步模型（Sync）                                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户请求 → 处理任务 → 返回响应                             │
│      ↑                                                    │
│      └── 等待任务完成（阻塞）                               │
│                                                             │
│  优点：                                                     │
│  - 简单，容易理解                                          │
│  - 立即得到结果                                            │
│                                                             │
│  缺点：                                                     │
│  - 任务耗时影响响应                                        │
│  - 并发能力差                                              │
│  - 资源利用率低                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  异步模型（Async）                                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户请求 → 立即返回 → 后台处理任务                         │
│      ↓                                                    │
│  用户可以继续其他操作                                       │
│      ↓                                                    │
│  任务完成 → 通知用户（轮询/推送）                           │
│                                                             │
│  优点：                                                     │
│  - 响应快                                                  │
│  - 并发能力强                                              │
│  - 资源利用率高                                            │
│                                                             │
│  缺点：                                                     │
│  - 复杂度提高                                              │
│  - 无法立即得到结果                                        │
│  - 需要回调机制                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 第二步：@Async注解

### 问题：如何让方法异步执行？

#### 代码实现：异步方法

```java
// service/OrderService.java
package com.taoshu.service;

import org.springframework.scheduling.annotation.Async;
import org.springframework.stereotype.Service;
import lombok.extern.slf4j.Slf4j;

@Service
@Slf4j
public class OrderService {

    /**
     * 下单（同步）
     */
    public Order createOrder(CreateOrderRequest request) {
        # 1. 创建订单
        Order order = buildOrder(request);

        # 2. 扣减库存
        decreaseStock(order);

        # 3. 清空购物车
        clearCart(order);

        # 4. 异步：发送通知
        sendOrderNotification(order);

        # 5. 异步：更新统计
        updateStatistics(order);

        return order;  # 立即返回，不等待异步任务
    }

    /**
     * 发送订单通知（异步）
     */
    @Async  # 关键注解！
    public void sendOrderNotification(Order order) {
        log.info("开始发送订单通知：{}", order.getOrderNo());

        try {
            # 发送短信
            smsService.send(order.getUserId(), "订单创建成功");

            # 发送邮件
            emailService.send(order.getUserId(), "订单详情", "...");

            # 通知商家
            notifyMerchant(order.getSellerId(), order);

            log.info("订单通知发送完成：{}", order.getOrderNo());

        } catch (Exception e) {
            # 异步任务失败，不影响主流程
            log.error("发送订单通知失败：{}", order.getOrderNo(), e);
        }
    }

    /**
     * 更新统计（异步）
     */
    @Async
    public void updateStatistics(Order order) {
        try {
            # 更新用户统计
            userService.updateOrderCount(order.getUserId());

            # 更新商家统计
            sellerService.updateSalesCount(order.getSellerId());

            # 更新商品统计
            bookService.updateSalesCount(order.getBookId());

        } catch (Exception e) {
            log.error("更新统计失败", e);
        }
    }
}
```

**执行流程**

```
时间线：
0ms:   用户请求
100ms: 创建订单完成
200ms: 扣减库存完成
300ms: 清空购物车完成
       → 返回响应给用户（总耗时300ms）
       → @Async线程启动
300ms: 异步线程1：发送短信（500ms）
300ms: 异步线程2：更新统计（200ms）
800ms: 异步线程1：完成
500ms: 异步线程2：完成

用户等待时间：300ms（同步需要2300ms）
提升：87%
```

## 第三步：线程池配置

### 问题：默认的异步线程池够用吗？

```
Spring Boot默认异步配置：
- 线程池：SimpleAsyncTaskExecutor
- 特点：每次执行创建新线程
- 问题：
  1. 没有线程复用，性能差
  2. 并发高时创建大量线程
  3. 可能导致OOM（内存溢出）

场景：
1000个下单请求/秒
→ 1000个异步任务/秒
→ 创建1000个线程！
→ 每个线程占用1MB内存
→ 总内存：1000MB！
→ 可能OOM！
```

### 代码实现：自定义线程池

```java
// config/AsyncConfig.java
package com.taoshu.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.scheduling.annotation.EnableAsync;
import org.springframework.scheduling.concurrent.ThreadPoolTaskExecutor;
import java.util.concurrent.Executor;
import java.util.concurrent.ThreadPoolExecutor;

@Configuration
@EnableAsync  # 启用异步
public class AsyncConfig {

    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();

        # 核心线程数（一直存活）
        executor.setCorePoolSize(10);

        # 最大线程数（核心+队列满时创建）
        executor.setMaxPoolSize(50);

        # 队列容量（核心线程忙时，任务放队列）
        executor.setQueueCapacity(200);

        # 线程名前缀
        executor.setThreadNamePrefix("async-task-");

        # 空闲线程存活时间（秒）
        executor.setKeepAliveSeconds(60);

        # 拒绝策略（队列满+线程满时的处理）
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());

        # 等待所有任务完成后关闭线程池
        executor.setWaitForTasksToCompleteOnShutdown(true);

        # 等待时间（秒）
        executor.setAwaitTerminationSeconds(60);

        executor.initialize();
        return executor;
    }

    /**
     * 发送通知专用线程池
     */
    @Bean(name = "notificationExecutor")
    public Executor notificationExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(20);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("notification-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}
```

**线程池参数详解**

```
核心线程数（CorePoolSize）：
- 一直存活的线程
- 建议：CPU密集型 = CPU核心数
- 建议：IO密集型 = CPU核心数 × 2

最大线程数（MaxPoolSize）：
- 核心线程 + 临时线程的总数
- 建议：根据任务耗时和响应时间要求确定
- 公式：Max = CPU核心数 / (1 - 阻塞系数)
  - CPU密集型：阻塞系数=0.1 → Max ≈ CPU核心数 × 1.1
  - IO密集型：阻塞系数=0.8 → Max ≈ CPU核心数 × 5

队列容量（QueueCapacity）：
- 核心线程忙时，任务存放队列
- 建议：根据内存和任务大小确定

拒绝策略（RejectedExecutionHandler）：
1. AbortPolicy（默认）：抛异常
2. CallerRunsPolicy：调用者线程执行
3. DiscardPolicy：直接丢弃
4. DiscardOldestPolicy：丢弃最老任务

推荐：CallerRunsPolicy（降级，不丢任务）
```

**使用指定线程池**

```java

    # ...
}
```

## 第四步：异步回调

### 问题：如何知道异步任务的结果？

```
场景：
用户下单后，需要知道支付结果
      → 异步调用支付宝查询
      → 需要获取查询结果
      → 更新订单状态

问题：
@Async方法返回void，无法获取结果

解决：
使用CompletableFuture
```

### 代码实现：异步回调

```java
// service/OrderService.java
@Service
public class OrderService {

    /**
     * 异步查询支付状态
     * @return CompletableFuture<String> 支付状态
     */
    @Async
    public CompletableFuture<String> queryPaymentStatusAsync(String orderNo) {
        return CompletableFuture.supplyAsync(() -> {
            try {
                # 模拟查询支付状态（耗时）
                Thread.sleep(2000);

                # 调用支付宝查询
                boolean paid = alipayUtil.queryPayment(orderNo);

                return paid ? "TRADE_SUCCESS" : "WAIT_BUYER_PAY";

            } catch (Exception e) {
                log.error("查询支付状态失败", e);
                return "ERROR";
            }
        });
    }

    /**
     * 使用异步回调
     */
    public void checkPaymentAndNotify(String orderNo) {
        # 1. 异步查询
        CompletableFuture<String> future = queryPaymentStatusAsync(orderNo);

        # 2. 回调处理
        future.thenAccept(status -> {
            log.info("订单{}支付状态：{}", orderNo, status);

            if ("TRADE_SUCCESS".equals(status)) {
                # 更新订单状态
                updateOrderStatus(orderNo, 1);

                # 发送通知
                notifyUser(orderNo);
            }
        });

        # 3. 异常处理
        future.exceptionally(throwable -> {
            log.error("查询支付状态异常", throwable);
            return null;
        });

        log.info("查询支付状态请求已提交，订单：{}", orderNo);
    }

    /**
     * 多个异步任务组合
     */
    public void checkMultiplePayments(List<String> orderNos) {
        # 1. 创建多个异步任务
        List<CompletableFuture<String>> futures = orderNos.stream()
            .map(this::queryPaymentStatusAsync)
            .collect(Collectors.toList());

        # 2. 等待所有任务完成
        CompletableFuture<Void> allOf = CompletableFuture.allOf(
            futures.toArray(new CompletableFuture[0])
        );

        # 3. 所有任务完成后执行
        allOf.thenRun(() -> {
            log.info("所有支付状态查询完成");

            # 处理结果
            futures.forEach(future -> {
                future.thenAccept(status -> {
                    # 处理每个订单的状态
                });
            });
        });
    }

    /**
     * 两个任务都完成才执行
     */
    public void completePayment(String orderNo) {
        # 任务1：查询支付状态
        CompletableFuture<String> queryFuture = queryPaymentStatusAsync(orderNo);

        # 任务2：查询订单信息
        CompletableFuture<Order> orderFuture = getOrderAsync(orderNo);

        # 两个任务都完成后执行
        queryFuture.thenCombine(orderFuture, (status, order) -> {
            if ("TRADE_SUCCESS".equals(status)) {
                order.setStatus(1);
                orderRepository.save(order);
                return true;
            }
            return false;
        });
    }

    @Async
    public CompletableFuture<Order> getOrderAsync(String orderNo) {
        return CompletableFuture.supplyAsync(() -> {
            Order order = orderRepository.findByOrderNo(orderNo).orElse(null);
            return order;
        });
    }
}
```

**CompletableFuture常用方法**

| 方法 | 说明 | 示例 |
|------|------|------|
| `supplyAsync` | 异步执行，有返回值 | `supplyAsync(() -> "result")` |
| `thenAccept` | 消费结果，无返回值 | `thenAccept(result -> log.info(result))` |
| `thenApply` | 转换结果 | `thenApply(result -> result.length())` |
| `thenCombine` | 组合两个任务 | `thenCombine(future2, (r1, r2) -> r1 + r2)` |
| `allOf` | 所有任务完成 | `allOf(future1, future2)` |
| `anyOf` | 任一任务完成 | `anyOf(future1, future2)` |
| `exceptionally` | 异常处理 | `exceptionally(e -> "default")` |

## 第五步：消息队列异步

### 问题：@Async有什么限制？

```
@Async的局限：
1. 应用重启，任务丢失
2. 无法跨服务通信
3. 重试机制需要自己实现
4. 没有消息持久化
5. 无法削峰填谷

场景：
用户注册后发送欢迎邮件
      → @Async异步发送
      → 应用重启，邮件未发送
      → 用户没收到欢迎邮件
      → 体验差

解决：使用消息队列（RabbitMQ/RocketMQ）
```

### 架构对比

```
┌─────────────────────────────────────────────────────────────┐
│  @Async异步模型                                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [主线程]                                                   │
│      ↓ 发送消息到线程池                                     │
│  [线程池] ─── → 执行任务                                    │
│      ↓                                                     │
│  完成                                                      │
│                                                             │
│  问题：                                                     │
│  - 应用重启，任务丢失                                      │
│  - 无法跨服务                                              │
│  - 没有持久化                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  消息队列异步模型                                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [生产者] ─── → [消息队列] ← ─── [消费者1]                  │
│                   │                   [消费者2]              │
│                   │                   [消费者3]              │
│                   ↓                                        │
│              (持久化)                                       │
│                                                             │
│  优点：                                                     │
│  - 消息持久化，不丢失                                      │
│  - 应用重启，消息还在                                      │
│  - 支持跨服务                                              │
│  - 削峰填谷                                                │
│  - 失败重试                                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## ⚠️ 异步最佳实践

### 1. 区分同步/异步

```java
# 应该异步：
- 发送短信/邮件
- 更新统计
- 生成报表
- 数据同步
- 第三方API调用

# 应该同步：
- 创建订单
- 扣减库存
- 支付操作
- 数据验证
```

### 2. 异步事务

```java
# 注意：@Async方法在事务外执行

@Transactional
public void createOrder() {
    # 主流程事务
    Order order = saveOrder();

    # @Async方法不在事务中
    asyncService.sendNotification(order);

    # 如果这里抛异常，订单回滚，但通知已发送
}

# 解决：事务提交后再异步
@Transactional
public void createOrder() {
    Order order = saveOrder();

    # 使用TransactionSynchronization
    TransactionSynchronizationManager.registerSynchronization(
        new TransactionSynchronization() {
            @Override
            public void afterCommit() {
                asyncService.sendNotification(order);
            }
        }
    );
}
```

### 3. 异步异常处理

```java
@Async
public CompletableFuture<String> asyncMethod() {
    try {
        # 业务逻辑
        return CompletableFuture.completedFuture("success");
    } catch (Exception e) {
        log.error("异步任务失败", e);
        return CompletableFuture.failedFuture(e);
    }
}
```

## 📊 Django vs Spring Boot 对比

| 功能 | Django | Spring Boot | 难度 |
|------|--------|-------------|------|
| 异步任务 | Celery | @Async / MQ | Celery更强大 |
| 线程池 | `ThreadPoolExecutor` | `ThreadPoolTaskExecutor` | 相当 |
| 异步回调 | `.get()` / `.then()` | `CompletableFuture` | Spring Boot更现代 |
| 消息队列 | Celery + RabbitMQ | RabbitMQ | 相当 |
| 定时任务 | `django-crontab` / Celery Beat | `@Scheduled` | Spring Boot更简单 |
| 任务监控 | Flower | Actuator | 相当 |

**Django优势**：
- Celery功能强大，生态完善
- Flower监控界面友好

**Spring Boot优势**：
- `@Async` 注解简单
- `CompletableFuture` 现代化
- Actuator自带监控

## 🎯 本章小结

| 知识点 | 为什么需要 | 怎么做 |
|--------|-----------|--------|
| **异步任务** | 提升响应速度，提高并发 | @Async注解 |
| **线程池** | 复用线程，避免OOM | ThreadPoolTaskExecutor |
| **CompletableFuture** | 异步回调，组合任务 | thenAccept / thenCombine |
| **消息队列** | 任务持久化，削峰填谷 | RabbitMQ |
| **重试机制** | 提高可靠性 | @Retryable / RabbitMQ retry |
| **事务同步** | 事务提交后执行 | TransactionSynchronization |

**记忆口诀**：
- **@Async异步快**，**线程池复用**
- **回调CompletableFuture**，**组合任务便**
- **简单用@Async**，**可靠用MQ**
- **事务外异步**，**同步要提交**
